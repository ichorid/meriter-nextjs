# План ручного юзер-тестирования перед продом — 2026-06-11

> **Совместный документ (развёрнутая инструкция для QA на dev):** [`QA-SHARED-DOCUMENT-RU.md`](./QA-SHARED-DOCUMENT-RU.md) — https://dev.meriter.pro, самостоятельная регистрация, Google Таблица.

Фичи: **(A)** закреплённый пост, **(B)** общий кошелёк сообщества с дочерними проектами, **(C)** совместный документ.

Среда: локальный запуск (web :8001, api :8002, Mongo rs0), `FAKE_DATA_MODE=true`, `TEST_AUTH_MODE=true`.

## Тестовые роли

| Роль | Как получаем |
|---|---|
| **Admin** (lead сообщества) | fake superadmin / создатель сообщества |
| **User1, User2** (участники) | fake users, вступают в сообщество |
| **Outsider** (не участник) | fake user без вступления |

Каждая роль — отдельный браузерный контекст/сессия (отдельные JWT cookie).

## A. Закреплённый пост

| # | Сценарий | Ожидание |
|---|---|---|
| A1 | Админ создаёт пост с галкой «Закрепить» | Пост создан, в ленте первым, бейдж «Закреплено» под шапкой |
| A2 | Админ редактирует обычный пост → закрепляет | Пост поднимается наверх ленты, бейдж появляется |
| A3 | Админ открепляет пост | Пост возвращается в обычный порядок, бейдж исчезает |
| A4 | Обычный участник создаёт/редактирует пост | Контрол закрепления **не виден** |
| A5 | Несколько закреплённых постов | Все вверху первой страницы; порядок детерминированный (свежие закреплённые выше); пагинация не дублирует и не теряет посты |
| A6 | Лента с пагинацией (много постов) | Закреплённые только на первой странице, обычная сортировка дальше не сломана |
| A7 | Закреплённый пост в других выдачах (хаб, профиль, биржа) | Закрепление влияет только на ленту сообщества; в остальных — обычный порядок, бейдж не мешает |
| A8 | API-защита: не-админ пытается закрепить (через devtools/tRPC) | Ошибка FORBIDDEN |

## B. Общий кошелёк (parent team + child projects)

| # | Сценарий | Ожидание |
|---|---|---|
| B1 | Тоггл «Общий кошелёк с проектами» в настройках сообщества | Виден админу обычного сообщества; у проекта — отсутствует |
| B2 | Включение тоггла при «свежих» дочерних проектах (нулевые балансы) | Успешно; CommunityWallet родителя создаётся |
| B3 | Включение при дочернем проекте с ненулевым community wallet или пользовательскими балансами | Ошибка с понятным сообщением, тоггл не включается |
| B4 | Привязка нового (пустого) проекта к родителю с общим кошельком | Успешно |
| B5 | Привязка проекта с балансом к такому родителю | Отказ (guard в link request / project.service) |
| B6 | Голос за пост в дочернем проекте | Заслуги ходят через кошелёк **родителя**: баланс юзера один и тот же в родителе и проекте |
| B7 | Доход автора поста в проекте | Зачисляется в кошелёк родителя |
| B8 | История заслуг (merit history) в родителе и проекте | Объединённая, транзакции из проекта видны, без дублей |
| B9 | Снятие/withdraw, перевод заслуг, опрос (poll), инвестиция в проекте | Все операции через shared root, балансы сходятся |
| B10 | Комиссии (создание поста, предложение варианта документа) | По-прежнему из глобального кошелька (fee → GLOBAL), shared wallet не вмешивается |
| B11 | Выключение тоггла | Операции снова идут через собственные кошельки; ничего не падает |
| B12 | Сообщество без проектов: включение/выключение | Работает, ошибок нет |

## C. Совместный документ

### C0. Создание и базовая структура

### C1. Прямые правки админа (lead save)
| # | Сценарий | Ожидание |
|---|---|---|
| C1.1 | Админ правит текст напрямую (unified editor) и сохраняет | Официальный текст обновлён сразу, без голосования; зеркало `futureVisionText`/`description` обновлено |
| C1.2 | Структурные правки: новый абзац, заголовок, список, удаление абзаца | Сохраняются корректно, id блоков стабильны |
| C1.3 | Прямая правка при наличии открытых предложений | Не ломает открытые threads; предупреждение/корректное поведение (rebase подсветок) |
| C1.4 | История блока | Записи с reason `admin`/`initial`, превью старого и нового текста |

### C2. Предложения правок
| # | Сценарий | Ожидание |
|---|---|---|
| C2.1 | **Админ** в режиме «Предложить» создаёт правку | Создаётся вариант + voting thread, официальный текст НЕ меняется, комиссия списана |
| C2.2 | **User1** предлагает правку (replace внутри абзаца) | Вариант создан, fee списан и сгорел, подсветка (жёлтая) видна всем, rail показывает предложение |
| C2.3 | User1 предлагает добавление абзацев в конец | insert_after patch; в превью видны новые абзацы; official не изменён |
| C2.4 | User1 предлагает удаление блока | В подсветке — зачёркнутый блок |
| C2.5 | User2 предлагает правку, пересекающуюся с открытой | Слияние в тот же thread, срок продлён, toast «Правка добавлена в текущее голосование…» |
| C2.6 | User2 предлагает правку в другом месте | Отдельный thread, отдельная волна |
| C2.7 | Повторное предложение того же юзера на тот же блок при открытом его варианте | Отказ «уже есть открытый вариант» |
| C2.8 | Недостаточно заслуг на комиссию | Кнопка неактивна / ошибка INSUFFICIENT_FUNDS, вариант не создан |
| C2.9 | Отзыв варианта автором | Статус withdrawn, деньги не возвращаются; если вариантов не осталось — волна закрывается |
| C2.10 | Outsider пытается предложить | Запрет |

### C4. Закрытие голосования
| # | Сценарий | Ожидание |
|---|---|---|
| C4.1 | **Раннее закрытие** админом, режим «по голосам» (by_votes) | Волна закрыта немедленно, лидер с rating>0 применяется к официальному тексту, остальные closed-not-winner |
| C4.2 | Раннее закрытие при rating ≤ 0 | Победителя нет, официальный текст не меняется |
| C4.3 | **Плановое закрытие** (дедлайн волны, cron) в manual-режиме | Лидер → closed-winner «ожидает применения»; админ видит кнопку «Применить»; после применения — official обновлён |
| C4.4 | Плановое закрытие в auto-режиме | Победитель применяется автоматически кроном, плашка «принято автоматически» |
| C4.5 | **Принудительное закрытие**: force_official | Все варианты закрыты не-победителями, официальный текст сохранён |
| C4.6 | Принудительное закрытие: force_variant (волевой выбор варианта) | Выбранный вариант применён с reason admin |
| C4.7 | Stale apply: официальный текст изменился между предложением и применением | Запрос подтверждения (confirmStale), без молчаливой порчи текста |
| C4.8 | После закрытия волны новое предложение | Открывается новая волна/thread корректно |
