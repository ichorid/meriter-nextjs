## 1. Зачем это руководство

«Совместный документ» — способ редактировать общий текст сообщества (например, **Образ будущего** команды или **Описание** проекта) не напрямую, а через **предложения** и **голосование заслугами**.

Это руководство поможет:

- понять, что именно проверять;
- пройти сценарии шаг за шагом;
- одинаково записывать результаты и сообщать об ошибках.

Специальных знаний программирования не нужно. Достаточно браузера, двух аккаунтов (лучше три) и внимательности.

## 2. Как устроена проверка

### 2.1. Обозначения в кейсах


| Значок | Смысл                                                |
| ------ | ---------------------------------------------------- |
| **P0** | Обязательно проверить — без этого фича «не работает» |
| **P1** | Важно, но можно после P0                             |
| **P2** | По возможности                                       |
| ✅      | Кейс пройден                                         |
| ❌      | Найдена ошибка                                       |
| ⏭      | Пропуск (нет условий / не успели)                    |


### 2.2. Роли в тестах (не «профили людей», а права в системе)


| Роль         | Кто это                                                           | Что обычно может                                                                        |
| ------------ | ----------------------------------------------------------------- | --------------------------------------------------------------------------------------- |
| **Лид**      | Создатель сообщества или назначенный администратор                | Редактировать документ напрямую, закрывать голосование, настройки, закреплять фрагменты |
| **Участник** | Вступил в сообщество                                              | Читать, предлагать правки, голосовать                                                   |
| **Гость**    | Зарегистрирован на платформе, но **не** состоит в этом сообществе | Только читать документ (если открыт); предложения и голоса — нет                        |


Для одного сценария «два человека одновременно» нужны **два браузера** (или обычное окно + режим инкognito) под **разными** аккаунтами.

### 2.3. Что считать ошибкой


| Уровень       | Когда ставить                                         | Пример                                                    |
| ------------- | ----------------------------------------------------- | --------------------------------------------------------- |
| **Блокер**    | Нельзя продолжить сценарий                            | Кнопка «Отправить» не работает, страница падает           |
| **Серьёзный** | Функция есть, но результат неверный или сильно путает | Голос прошёл без комментария, хотя комментарий обязателен |
| **Мелкий**    | Неудобство, опечатка, кривой отступ                   | Обрезана кнопка на телефоне                               |
| **Идея**      | Не ошибка, а пожелание                                | «Хотелось бы крупнее шрифт»                               |


### 2.4. Как описать ошибку (шаблон)

Запишите **одним сообщением**:

1. **ID кейса** (например, DOC-E2)
2. **Что делали** — по шагам, какие кнопки нажимали (названия с экрана)
3. **Что ожидали**
4. **Что получилось** на самом деле
5. **Скриншот** или короткое видео (ссылка)
6. **Браузер и устройство** (Chrome / Safari, Windows / Android / iPhone)
7. **URL страницы** (скопировать из адресной строки)
8. **Под какой ролью** были (Лид / Участник / Гость)

**Плохо:** «Голосование сломано».

**Хорошо:** «DOC-E2, Участник, Chrome Android. Открыл документ → выбрал предложение → «Поддержать» → поле комментария пустое → нажал отправить. Ожидал: блокировку. Получил: голос засчитан, рейтинг +1. Скрин: …»

### 2.5. Браузеры и устройства

Минимум:

- **Компьютер:** Chrome или Edge, ширина окна «на весь экран»
- **Телефон** или эмуляция в браузере (F12 → режим телефона, ширина ~375 px)

Желательно хотя бы один раз прогнать все **P0** на телефоне.

### 2.6. Термины (как на экране)


| Термин                                       | Простыми словами                                             |
| -------------------------------------------- | ------------------------------------------------------------ |
| **Заслуги**                                  | Внутренняя «валюта» платформы (в интерфейсе **не** «мериты») |
| **Официальный текст** / **Исходный вариант** | Текст, который сейчас считается согласованным                |
| **Предложение** / **Предложить правку**      | Ваш вариант изменения; сам по себе текст не меняет           |
| **Голосование** / **Волна**                  | Период, когда по предложениям можно голосовать               |
| **Применить** / **Выбрать победителем**      | Записать победившее предложение в официальный текст          |
| **Закрепить**                                | Лид запрещает правки в выделенном фрагменте                  |

## 3. Подготовка: аккаунты и тестовое сообщество

### 3.1. Регистрация

1. Откройте [https://dev.meriter.pro](https://dev.meriter.pro)
2. Войдите или зарегистрируйтесь (обычно **по email** — ссылка приходит на почту)
3. Заполните профиль (имя), если система попросит

**Рекомендуется три аккаунта:**


| Аккаунт | Назначение                       |
| ------- | -------------------------------- |
| **A**   | Лид тестового сообщества         |
| **B**   | Участник                         |
| **C**   | Гость (не вступает в сообщество) |


Можно завести их на разные email или попросить коллегу держать второй аккаунт.

### 3.2. Заслуги на балансе

Для предложений и голосов нужны заслуги. После регистрации на dev часто начисляются **приветственные заслуги** на глобальный кошелёк.

Проверьте в **Профиле** / разделе баланса, что есть хотя бы **10–20 заслуг** на каждом аккаунте A и B. Если мало — сообщите команде разработки (на dev иногда меняют настройки).

### 3.3. Создание тестового сообщества (аккаунт A)

1. Под аккаунтом **A** найдите **«Создать сообщество»** (в профиле или боковом меню)
2. Заполните форму:
  - Название, например: `QA Документ тест`
  - В блоке **Образ будущего** напишите несколько абзацев текста (это станет основой совместного документа)
3. Сохраните сообщество

> **Если кнопка «Создать сообщество» недоступна** (сообщение «только организаторы и лиды»): договоритесь внутри команды, кто уже имеет право создавать команды, или попросите доступ у администраторов dev. Без своего сообщества полный цикл P0 не пройти.

### 3.4. Пригласить участника B

1. Аккаунт **A** → страница сообщества → **Участники** / **Пригласить**
2. Скопируйте **ссылку приглашения от администратора** (по ней вступление без заявки)
3. Откройте ссылку в другом браузере под аккаунтом **B** → **Вступить**

Аккаунт **C** в сообщество **не** вступает.

### 3.5. Где открыть документ

После создания сообщества:

- На странице сообщества — ссылка **«Открыть документ»** или вкладка **«Документы»**
- Прямой путь: `https://dev.meriter.pro/meriter/communities/{id_сообщества}/documents` → открыть документ типа **«Образ будущего»**

Для проверки **описания проекта** (P1): создайте **проект** внутри команды — появится документ **«Описание»** с тем же принципом работы.

### 3.6. Ускорение голосования (для лидов)

Чтобы не ждать часами окончания волны:

1. Лид → документ → **«Настройки документа»**
2. **«Длительность голосования»** — поставьте **1** час (минимум)
3. Досрочное закрытие: **«Закрыть голосование»** → **«По набранным баллам»**

### 3.7. Порядок прогона

1. Блок **A** — вход и навигация
2. Блок **B** — чтение
3. Блок **C** — правки лида
4. Блок **D** — предложения
5. Блок **E** — голосование
6. Блок **F** — закрытие и применение
7. Блок **G** — синхронизация и уведомления
8. Блок **H** — настройки
9. Блок **I** — внешний вид
10. Блок **J** — дополнительно

## 4. Кратко о фиче

### 4.1. Как это работает

```
Читаем документ
    → выделяем фрагмент текста
    → «Предложить правку»
    → платим комиссию (заслуги сгорают)
    → другие голосуют «за» / «против» с комментарием
    → лид закрывает голосование
    → победившая правка попадает в официальный текст
```

**Лид** может менять текст **сразу** (кнопка **«Сохранить»** в редакторе), минуя голосование.

**Участник** официальный текст **не** правит напрямую — только через предложения.

### 4.2. Экран документа

**На компьютере (широкий экран):**

- По центру — текст документа
- Справа — панель **«Предложения»**
- Сверху — название, **«Настройки документа»** (лид), **«Сохранить»** (лид)

**На телефоне:**

- Текст на весь экран
- Внизу — **«Предложения по правкам (N)»** (открывает список)

### 4.3. Комиссии

- **Предложение правки** — списание заслуг при отправке; **не возвращаются** при отзыве
- **Голос «за»** — сначала дневная квота (если есть), потом кошелёк
- **Голос «против»** — только с кошелька
- Комиссия за предложение списывается с **глобального** кошелька заслуг

## 5. Карта проверки — кейсы

Формат каждого кейса:

- **Кто** — Лид / Участник / Гость  
- **Шаги** — что нажать  
- **Ожидание** — что должно произойти  
- **Запись** — что отметить в таблице

## 7. Что не входит в эту проверку

- Создание **произвольных** документов (режим «все документы») — в MVP не включено
- Telegram-бот и мини-приложение
- Автоматическое ожидание cron «ровно 5 минут» — используйте **«Закрыть голосование»** вручную
- Проверка безопасности через инструменты разработчика (API, консоль)
- Импорт из Google Docs, вставка картинок в документ
- Откат истории к старой версии одной кнопкой
