# Отчёт для коллеги: технические сведения по обработке ПД (Meriter / «Мультиобраз»)

**Источник:** анализ репозитория `meriter-nextjs` (без доступа к прод-конфигам, договорам и юр. лицу).  
**Назначение:** заполнение уведомления/формы РКН и внутренняя сводка по ИС.

---

## 1. Scope: что этот отчёт покрывает и чего в нём нет

**Покрывает:** как устроена обработка данных в коде и типовой инфраструктуре (каналы, хранилища, категории данных, интеграции, меры из документации репозитория).

**Не покрывает (нужно брать у оператора / из договоров / инфраструктуры):** регион обработки как юридическая география; точное местонахождение БД и адрес ЦОД; реквизиты организации и контрагентов по хранению; дата начала обработки; сведения о лицах с доступом к ПД в ГИС МЭД (если применимо — это не следует из кода); класс СКЗИ, наименование, изготовитель, серийные номера средств шифрования в смысле реестра ФСБ; финальное решение о трансграничной передаче и перечень стран — только по фактическим договорам и размещению сервисов.

---

## 2. Продуктовая связка Meriter и «Мультиобраз»

- **«Мультиобраз»** — не отдельный репозиторий/бэкенд, а **пилотный режим** того же приложения: флаги `PILOT_MODE` (API) и `NEXT_PUBLIC_PILOT_MODE` (web), плюс id hub-сообщества (`PILOT_HUB_COMMUNITY_ID` / `NEXT_PUBLIC_PILOT_HUB_COMMUNITY_ID`).
- **Те же** сборки образов, MongoDB, JWT, S3, OAuth, Telegram и т.д.; отличия — **маршруты UI**, **ограничения части API** и **дополнительные поля** у сущностей «мечта» (например `pilotMeta`, `pilotDreamRating` в модели сообщества).
- **Вывод для формы:** инфраструктура и каналы обработки ПД **одинаковые**; различать цели/сценарии (полная платформа vs пилот) и фактический набор собираемых полей на конкретном стенде.

---

## 3. Архитектура потока данных (технический путь)

1. Клиент: браузер или Telegram Mini App (при включённой интеграции).
2. Транспорт: **HTTPS** (прокси **Caddy**: заголовки безопасности, HSTS в проде — см. `Caddyfile`, `SECURITY.md`).
3. Backend: **NestJS**, основной API — **tRPC** (`/trpc`), дополнительно **REST** `/api/v1/...` (auth, часть legacy и служебных потоков).
4. Хранение: **MongoDB** (основная БД приложения); файлы — **S3-совместимое хранилище** (endpoint/регион задаются переменными окружения).
5. Сессия: **JWT**, домен cookie привязан к `DOMAIN` (см. `api/env.example`).

---

## 4. Цели обработки (логика системы, не юридическая формулировка)

Поддерживаются сценарии, под которые в коде есть функционал (формулировки для РКН — за юристом/оператором):

- регистрация и вход, ведение учётной записи;
- профиль и пользовательский контент (публикации, обсуждения, голоса, опросы и т.д.);
- внутренняя «экономика» заслуг (кошельки, квота, транзакции, переводы между пользователями в контексте сообществ/проектов — по доменной логике);
- уведомления;
- администрирование и роли (в т.ч. superadmin);
- при включении — мониторинг ошибок (**Sentry**: клиент и сервер при наличии DSN);
- при включении — Telegram-бот и связка с Telegram Web Auth (переменные и флаги в `env.example`).

## 5. Категории персональных данных (по моделям и потокам)


| Категория                              | Где в системе                                                                                                                                                   |
| -------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Идентификаторы аккаунта                | `users`: `id`, пара `authProvider` + `authId`                                                                                                                   |
| Имя / отображаемое имя                 | `displayName`, опционально `firstName`, `lastName`, `username`                                                                                                  |
| Контакты в профиле                     | `profile.contacts.email`, `profile.contacts.messenger` (в shared-схеме указано ограничение видимости для части ролей — реализация прав на стороне API/UI)       |
| Локация                                | `profile.location.region`, `profile.location.city` (в схеме зафиксировано происхождение региона/города через OSM API при заполнении)                            |
| Прочее в профиле                       | `bio`, `about`, `website`, `educationalInstitution`, `avatarUrl`                                                                                                |
| Данные для входа по SMS                | коллекция OTP: **номер телефона** (формат E.164), код OTP, служебные поля (`sms-otp.schema.ts`)                                                                 |
| Данные для входа по email / magic link | отдельные коллекции OTP и magic link; у magic link — канал `sms` | `email`, поле **цели** (`target`) как телефон или email                                      |
| WebAuthn (passkeys)                    | `users.authenticators`: идентификатор учётных данных, **публичный** ключ, счётчик, тип устройства и др. — не пароль в открытом виде                             |
| Контент и активность                   | публикации (текст, медиа URL, метаданные событий и пр.), комментарии/голоса, избранное, уведомления, записи кошелька/транзакций — по соответствующим коллекциям |
| Технические / журналы                  | при неуспешной аутентификации — **IP**, путь, усечённый User-Agent (`SECURITY.md`)                                                                              |
| Ошибки клиента/сервера                 | при включённом Sentry — возможна отправка стектрейсов и контекста (настройка фильтрации чувствительных данных — операционная задача)                            |


Отдельно: персональные данные в **свободном тексте** публикаций и комментариев пользователь вносит самому себе — учитывать при оценке состава ПД.

## 6. Категории субъектов

В коде моделируются **пользователи платформы**. Продуктовая документация MVP указывает аудиторию «школьники», но **автоматической** маркировки возраста/статуса ребёнка в схемах не видно — квалификация субъектов для РКН остаётся за оператором и фактическими условиями доступа.

---

## 7. Операции с персональными данными

Характерные для такой ИС: сбор через формы/API, запись и хранение в MongoDB и S3, использование в бизнес-логике, передача по сети, изменение (редактирование профиля и контента), удаление (в т.ч. мягкое для части сущностей по доменным процедурам). Обработка в основном **автоматизированная**; доступ администратора возможен через предусмотренные роли.

---

## 8. Криптография и защита каналов (без реестра СКЗИ)

Из репозитория следует использование:

- **TLS (HTTPS)** для взаимодействия с пользователем.
- **JWT** для сессии (секрет `JWT_SECRET`).
- **Аутентификация к MongoDB** (пароли пользователей БД в типовом `docker-compose`).
- **WebAuthn:** хранение данных для проверки подлинности в формате стандарта (публичные ключи и метаданные), не «хранение пароля соцсети».

**Не документировано в репозитории:** шифрование полей на уровне приложения в БД; использование конкретных сертифицированных СКЗИ из реестра с классами и серийниками — для граф РКН нужны **закупочные/внедренческие** документы, не код.

---

## 9. Трансграничная передача и внешние операторы (технические риски)

Код **не заменяет** юридический анализ, но фиксирует интеграции, которые при боевом включении обычно требуют учёта:


| Канал                                                            | Зачем в продукте                                  |
| ---------------------------------------------------------------- | ------------------------------------------------- |
| OAuth-провайдеры (в т.ч. Google и др.)                           | вход                                              |
| Telegram (Bot API, Web Auth, Mini App)                           | вход/бот/виджеты по конфигу                       |
| Sentry                                                           | мониторинг ошибок                                 |
| S3-совместимое API                                               | файлы; регион и юрисдикция — по выбранному облаку |
| Внешние origin в CSP (например Google Fonts, `api.telegram.org`) | загрузка ресурсов/запросы к API                   |


Итог для коллеги: перечень стран и операторов трансграничной передачи — из **фактических договоров, регионов облака и политик провайдеров**, не из git.

---

## 10. Меры по безопасности ПД (как описано в репозитории)

`SECURITY.md` и конфигурация Caddy/Docker: заголовки безопасности (HSTS, CSP, XFO, Referrer-Policy и др.), ограничение экспозиции MongoDB наружу в типовом compose, сегментация сетей Docker, лимиты ресурсов, работа процессов от non-root / read-only где задано, список чувствительных переменных (JWT, пароли БД, S3, OAuth, токен бота), логирование неуспешных входов с IP. Это **описание реализованных мер**, не полный комплаенс-набор 152-ФЗ без организационной части.

---

## 11. Разделение для формы: Meriter vs «Мультиобраз»


| Аспект                      | Meriter (основной режим)                 | «Мультиобраз» (пилот)                                                                   |
| --------------------------- | ---------------------------------------- | --------------------------------------------------------------------------------------- |
| Инфраструктура              | Один стек (API, MongoDB, S3, Caddy, JWT) | То же                                                                                   |
| Сценарии                    | Полный функционал платформы              | Упрощённые маршруты (`/`, `/create`, `/profile` и т.д.), часть мутаций отключена на API |
| Специфические данные        | —                                        | Доп. поля пилотских сущностей (например агрегаты рейтинга мечты, `pilotMeta`)           |
| Учётная запись пользователя | Общая модель `users`                     | Та же модель                                                                            |


Для коллеги: если пилот и основной продукт идут **в одной БД и на одних серверах**, блоки про **место хранения БД, ЦОД, транспортное шифрование** совпадают; различаются **описание целей/объёма обработки** под пилот.

---

## 12. Что передать коллеге как чек-лист «вне репозитория»

- Юридическое лицо оператора и основание обработки.
- Регион(ы) обработки и фактическое размещение серверов.
- Адрес ЦОД / площадки размещения БД (если требуется формой).
- Дата начала обработки.
- Полный список трансграничных получателей и правовое основание передачи.
- Документы по СКЗИ (если заявляются реестровые средства).
- Регламент доступа к ПД и список лиц (включая обработчиков по договору и отдельную панель по ГИС МЭД — если применимо).

---

**Итог одной строкой:** технически это одна ИС с MongoDB и S3, JWT и HTTPS, множеством способов входа и богатым пользовательским контентом; пилот «Мультиобраз» меняет **сценарии и часть полей**, но не отдельную инфраструктуру; юридически значимые поля формы РКН про географию, даты, СКЗИ и трансграничность код не заменяет.
