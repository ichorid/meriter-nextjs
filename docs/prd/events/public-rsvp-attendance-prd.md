## 1. Резюме для стейкхолдеров

Нужно (а) показывать календарь/списки ивентов и карточки ивентов **всем авторизованным пользователям**, даже без членства в команде/проекте; при попытке отметить «пойду» — предлагать вступление с заявкой и автотекстом. (б) поверх существующего RSVP ввести **учёт явки**: «пришёл» / «не пришёл», только вручную или по QR; админы ивента могут менять статус любому участнику из списка «пойду».

Текущая реализация (ориентир для миграции):

- Бэкенд: `assertCanViewEventsInCommunity` в `events.router.ts` запрещает `getEventsByCommunity` не-участникам для `isProject` и `typeTag === 'team'`.
- RSVP: `eventAttendees: string[]` на публикации; `EventService.attendEvent` требует роль в `communityId`.
- Фронт: ссылки «Ивенты» в `CommunityPageClient` / `ProjectPageClient` только при `isCommunityMember` / `isMember`; лента ивентов падает в ошибку при `FORBIDDEN` на `getEventsByCommunity`.
- Параллельный путь: `attendViaInvite` и **EventInvite** — гость может попасть в участники **без** членства в сообществе (см. раздел 6 — конфликт с новой политикой).

## 2. Цели и метрики успеха

| Цель | Измеримость (после релиза) |
|------|---------------------------|
| Не-участник видит афишу ивентов команды/проекта | E2E: гость открывает `/meriter/communities/:id/events` и `/meriter/projects/:id/events`, видит список |
| Снижение трения для «хочу прийти» | Конверсия: клик «пойду» → открыт диалог → отправка заявки |
| Прозрачная явка на месте | Админ отмечает приход за &lt; N секунд (ручная кнопка или один скан QR) |

Нефункциональные: безопасность QR (подделка, replay), i18n (RU/EN минимум), доступность (крупная кнопка QR, контраст).

## 3. Область работ (Scope)

### 3.1 In scope

1. **Публичный просмотр** ленты ивентов и навигация к разделу для любого авторизованного пользователя (для команд/проектов — ранее было закрыто).
2. **Диалог при RSVP** не-участника: текст про необходимость членства + CTA «Подать заявку» / «Отмена».
3. **Связка заявки во вступление** с автокомментарием и **авто-RSVP после одобрения** лидом.
4. **Модель явки** + API + UI для админов ивента и для участника (QR).
5. Обновление **business-events** rule, **CHANGELOG** (или принятый в репо формат релиз-нот), **тесты** (unit + e2e по возможности).

### 3.2 Out of scope (явно отложить или отдельный тикет)

- Нативное приложение сканера (только web; см. UX сканера).
- Автоматическое «не пришёл» по таймауту после конца ивента.
- Массовый импорт явки из CSV.
- Push-напоминания «покажи QR».
- Изменение правил **мерит-переводов** для гостей по инвайту (если политика RSVP для гостей изменится — отдельное решение).

## 4. Персоны и сценарии

| Персона | Сценарий |
|---------|----------|
| Гость (не в команде) | Открыл страницу команды → «Ивенты» → видит список → «Пойду» → диалог → заявка или отмена |
| Участник | Как сейчас: «Пойду» / «Не пойду»; после «Пойду» — кнопка **показать QR** |
| Лид / админ ивента | Список участников с колонками RSVP + явка; ручное переключение; режим «Сканировать QR» |
| Суперадмин | Те же права, что у лида на отметку явки (уточнить в реализации через `PermissionService` / существующие проверки `edit` публикации) |

**Определение «администратор ивента» (нормативно для реализации):**

- Пользователь, которому разрешено действие **`edit`** (и при необходимости **`delete`**) над **данной публикацией-ивентом** через существующий `checkPermissionInHandler` / правила сообщества для публикаций.
- Если в коде сейчас только автор может редактировать — **продуктово** нужно согласовать: лид сообщества = админ явки или только автор ивента. **Рекомендация в ТЗ:** считать админом явки того, кто может **редактировать событие** (как минимум автор + лиды по политике сообщества, если так уже заведено для `postType === 'event'`). Зафиксировать выбранное правило в `business-events.mdc`.

## 5. Функциональные требования

### 5.1 Публичный доступ к данным и навигации

1. **`events.getEventsByCommunity`**  
   - Для **любой** существующей `communityId`, если пользователь авторизован: возвращать список ивентов (upcoming/past), **без** требования роли в команде/проекте.  
   - Сохранить проверку «сообщество существует».  
   - Опционально: скрывать **внутренние** поля в ответе для гостя (например, PII в описании не трогаем; но список `eventAttendees` может быть сокращён — см. 5.4 приватность).

2. **Страница одного ивента** (`EventPage`, маршрут из `routes.eventView`)  
   - Убедиться, что не-участник может **читать** метаданные ивента (если сейчас где-то режется доступ — выровнять с политикой «афиша публична для авторизованных»).

3. **Фронт: ссылки «Ивенты»**  
   - `CommunityPageClient`, `ProjectPageClient`: показывать вход в раздел ивентов **всем авторизованным** (не только `isMember`), либо всем посетителям страницы — в зависимости от общей политики гостевого просмотра команды (сейчас страница команды может быть доступна не-участнику — проверить и согласовать).

### 5.2 RSVP не-участника → заявка + автокомментарий + отложенный RSVP

**Триггер:** пользователь без роли в `communityId` ивента нажимает «Пойду» (или эквивалент на карточке/странице).

**UI (модальное окно / bottom sheet на мобиле):**

- Заголовок: нейтральный, без юридического жаргона.
- Текст (RU, пример смысла): *«Чтобы принять участие в ивенте, нужно быть участником сообщества. Подать заявку на вступление? В заявку будет добавлен комментарий с названием ивента.»*
- Кнопки: **«Да, подать заявку»** | **«Нет»** (возврат к предыдущему состоянию без побочных эффектов).

**Действие «Да»:**

1. Вызвать существующий поток **`teams.submitTeamRequest`** (или `project.joinProject` для проектов — использовать тот же `TeamJoinRequestService.submitRequest` под капотом, как в `ProjectService.joinProject`).  
2. В `applicantMessage` передать **стабильный шаблон**, например:  
   `Хочу участвовать в ивенте: «{eventTitle}» (id: {publicationId})`  
   — чтобы лид видел контекст и можно было поддержать вручную. Ограничение **500 символов** — укорачивать title при необходимости.

**Отложенный RSVP:**

- При успешной отправке заявки **не** добавлять пользователя в `eventAttendees` сразу.
- Сохранить намерение «после вступления — автоматически отметить пойду». Варианты хранения (выбрать один на ревью кода):
  - **A (предпочтительно):** расширить схему `TeamJoinRequest` опциональными полями `pendingEventPublicationId?: string`, `pendingEventCommunityId?: string` (дубль для валидации).
  - **B:** отдельная коллекция `event_rsvp_pending` с TTL/очисткой при отклонении заявки.

**При одобрении заявки** (`TeamJoinRequestService.approveRequest` или единая точка после добавления роли):

- Если у заявки есть `pendingEventPublicationId` и публикация всё ещё `postType === 'event'` и не удалена: вызвать ту же логику, что `attendEvent` (идемпотентно `$addToSet`).
- Очистить pending-поля у заявки / удалить pending-запись.

**Отклонение / отзыв заявки:**

- Снять pending; пользователь **не** попадает в `eventAttendees`.

**Конфликты:**

- У пользователя уже есть **pending** заявка в тот же комьюнити без поля ивента: **Решение ТЗ:** при новом клике «Пойду» либо (1) показать «заявка уже на рассмотрении» + кнопка «добавить в комментарий лиду напоминание про ивент» (усложнение), либо (2) **обновить** `applicantMessage`/`pendingEventPublicationId` на сервере одной мутацией «append intent» (предпочтительно проще для UX: один запрос `teams.attachEventIntentToPendingRequest`). Минимальный MVP: если pending уже есть — показать тост и не создавать вторую заявку; опционально отдельный follow-up тикет на attach.

### 5.3 Модель данных: RSVP + явка

**Текущее:** `eventAttendees: string[]`.

**Целевое (пример нормализации):**

Заменить на массив объектов, например:

```ts
eventParticipants?: Array<{
  userId: string;
  /** зафиксированное намерение прийти; false = явно "не пойду" или отмена — уточнить семантику */
  rsvpGoing: boolean;
  /** null = ещё не отмечено; 'checked_in' | 'no_show' только руками админа */
  attendance: null | 'checked_in' | 'no_show';
  /** опционально: кто и когда отметил (аудит) */
  attendanceUpdatedAt?: Date;
  attendanceUpdatedByUserId?: string;
}>;
```

**Семантика статусов (согласно запросу заказчика):**

| Пользователь видит | Внутренне |
|--------------------|-----------|
| Пойду | `rsvpGoing === true` |
| Не пойду | `rsvpGoing === false` **или** отсутствие записи — **нужно выбрать одно** (рекомендация: явная запись только для «пойду»; «не пойду» = удалить из массива или `rsvpGoing: false` без права на QR — см. ниже) |
| Пришёл | `attendance === 'checked_in'` |
| Не пришёл | `attendance === 'no_show'` (только вручную админом) |

**Правила:**

- Поле **явки** имеет смысл только если `rsvpGoing === true` (для UI можно показывать «—» для не идущих).
- Смена RSVP с «пойду» на «не пойду» **сбрасывает** `attendance` в `null` и **инвалидирует** старый QR (новый токен при следующем «пойду»).

**Миграция БД:**

- Скрипт миграции: для каждой публикации с `eventAttendees: string[]` создать объекты `{ userId, rsvpGoing: true, attendance: null }`.

### 5.4 QR «билет» участника

**Условие выдачи:** пользователь в `rsvpGoing === true` для данного ивента.

**Содержимое QR (рекомендация безопасности):**

- Не класть в QR только сырой `publicationId` + `userId`.  
- Использовать **короткоживущий подписанный токен** (HMAC-SHA256 или JWT с секретом приложения) с полями: `publicationId`, `userId`, `iat`, `exp` (например 48ч или до `eventEndDate + 1d`).  
- Альтернатива: opaque `ticketId` в коллекции `event_checkin_tickets` с one-time флагом — дороже по IO, проще ротация.

**Процедуры tRPC (черновик контракта):**

- `events.getMyCheckInPayload` → `{ token, expiresAt }` (или URL для QR). Только для текущего пользователя и только если `rsvpGoing`.
- `events.checkInByToken` → input: `token` string. Вызывает **только** пользователь с правом админа ивента. Идемпотентно: если уже `checked_in`, success. Валидирует подпись, срок, соответствие ивенту. Устанавливает `attendance: 'checked_in'`. **Без** дополнительного confirm-диалога на сервере (как просили).

**Сканирование на фронте админа:**

- Кнопка «Сканировать QR» → доступ к камере (`getUserMedia`) + библиотека чтения QR (например `qr-scanner` или `@zxing/browser`) — зависимость и HTTPS требования задокументировать.  
- Fallback: ручной ввод токена в поле (для десктопа без камеры).  
- После успешного `checkInByToken` — тост «Участник отмечен как пришедший» + обновление списка.

**Участник:**

- На странице ивента и/или в модалке: заметная кнопка **«Показать QR участника»** (иконка `QrCode` + текст на мобиле).  
- Отображение крупного QR + подпись «{Имя ивента}» + таймер истечения (опционально).

### 5.5 Ручная отметка явки

- В UI списка участников (только для админа ивента): для каждой строки с `rsvpGoing` — селект или два действия: «Отметить пришёл» / «Отметить не пришёл» / «Сбросить явку» (сброс опционален, но полезен при ошибке).
- Вызов: `events.setParticipantAttendance` с `{ publicationId, targetUserId, attendance: null | 'checked_in' | 'no_show' }`.
- Сервер: проверка прав админа ивента; цель должна быть в участниках с `rsvpGoing === true`.

### 5.6 Приватность и отображение списков

**Решение для ревью продукта (закрепить в ТЗ перед релизом):**

- Вариант **P1 (строже):** не-участникам не отдаётся полный список ФИО/аватаров гостей — только **число** идущих.  
- Вариант **P2 (проще в MVP):** список идущих виден всем, кто видит ивент (как сейчас по сути для участников).  

Указать выбранный вариант в `business-events.mdc` и в UI.

## 6. Стык с существующими инвайтами (`attendViaInvite`)

**Исходная проблема:** инвайт позволял RSVP **без** членства в сообществе.

**Принятое решение (S-B, по [`agent-brief.md`](./agent-brief.md)):** RSVP по инвайту только у пользователя с **ролью** в `communityId` ивента. Лендинг инвайта ведёт на вступление в команду/проект; `getInvitePreview` возвращает **`isProject`** для корректной ссылки. `meritTransfer` с `eventPostId` требует получателя-**участника** контекста.

**Миграция старых данных** (RSVP без членства): не автоматизирована в релизе — см. бриф §1.5; при необходимости ops-скрипт.

## 9. Критерии приёмки (Acceptance Criteria)

1. Не-участник открывает `/meriter/communities/{teamId}/events` — видит upcoming/past без ошибки `FORBIDDEN`.  
2. Не-участник нажимает «Пойду» — видит диалог; «Нет» закрывает диалог, RSVP не меняется.  
3. «Да» создаёт заявку с текстом, содержащим название ивента; после approve лидом пользователь в `eventParticipants` с `rsvpGoing: true`.  
4. Участник с «Пойду» открывает QR; админ сканирует — статус «пришёл»; повторный скан не ломает (идемпотентность).  
5. Админ вручную ставит «не пришёл» — отображается в списке.  
6. ~~Инвайт-гость (S-A)~~ — **не актуально:** выбран **S-B**; RSVP по инвайту только после членства.  
7. i18n: нет сырых ключей в UI.
