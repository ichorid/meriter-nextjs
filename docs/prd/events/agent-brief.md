## 1. Зафиксированные решения (от стейкхолдера)

### 1.1 Кто может администрировать явку и сканировать QR

**Админ ивента** = любой из:

- автор публикации-ивента (`authorId`);
- пользователь с ролью **lead** в `communityId` этого ивента (и для проекта — лид проекта в том же `communityId`).

Ими вызываются мутации явки / `checkInByToken` / ручная смена «пришёл» / «не пришёл». Суперадмин — по общим правилам платформы (если уже есть обход для модерации — не расширять специально, если не требуется).

Реализация: одна функция вида `assertEventAttendanceAdmin(userId, publication)` в домене или тонком слое, не дублировать условия в трёх процедурах.

### 1.2 Видимость списка «пойду»

**Полная** для всех, кто видит страницу ивента / ленту (как сейчас для участников): те же поля профиля/аватаров, что и в текущем UI списка участников. Не режем список для гостей.

### 1.3 Семантика «не пойду» vs «не пришёл»

| Действие | Смысл | Данные |
|----------|--------|--------|
| Не пойду (сам участник) | отказ от RSVP | **удалить** пользователя из списка участников с намерением «пойду» (нет записи в «идущих») |
| Не пришёл (админ) | явка | **не удалять** из списка участников ивента; выставить поле явки `no_show` (или эквивалент из модели PRD) |

Участник **не** ставит «не пришёл» сам — только админ.

### 1.4 Когда участник **не может** менять свой RSVP («пойду» / «не пойду») и QR

Участник **заблокирован** на любые самостоятельные изменения RSVP (и отображение действий «пойду»/«не пойду» как недоступных), если выполняется **хотя бы одно** из условий:

1. **Ивент уже начался** — момент начала по данным публикации (`eventStartDate` + при наличии `eventTime` — по согласованной в коде одной схеме сравнения с «сейчас», с явным указанием таймзоны в комментарии кода и в `business-events.mdc`).
2. **Админ уже выставил явку** для этого пользователя: `checked_in` **или** `no_show` (в т.ч. ранняя отметка «пришёл» за 15 минут до старта).

Пока ивент не начался и явка у этого пользователя ещё `null` — участник может переключать «пойду» / «не пойду» (последнее = удаление из списка идущих).

**Сервер обязан** повторять те же правила: `attend` / `unattend` отклонять с понятной ошибкой при блокировке, не полагаться только на UI.

### 1.5 Инвайт по ссылке и членство

**Унификация под членство:** в конечном счёте **каждый**, кто числится участником ивента в смысле RSVP/явки/меритов в контексте ивента, должен быть **участником сообщества или проекта** (`communityId` ивента).

Следствия для кода:

- Поток **`attendViaInvite`**, который сегодня добавляет в `eventAttendees` **без** членства, **изменить или убрать**: недопустимо оставлять пользователя только «по инвайту» в списке ивента без роли в комьюнити.
- Рекомендуемая замена поведения (реализовать явно в ТЗ страницы инвайта):
  - страница инвайта показывает превью ивента и **призыв вступить** (заявка / принятие приглашения в команду — те же механики, что и для RSVP не-участника);
  - после того как пользователь стал участником комьюнити — **тогда** доступна кнопка «Пойду» (или автоматический `attend` — только если продуктово решите; по умолчанию — ручной «Пойду» после входа в комьюнити).
- Старые записи в БД «в eventAttendees, но не в комьюнити» — предусмотреть **миграцию или одноразовый скрипт** (например снять RSVP и уведомить лидов) либо задокументировать ручную чистку; минимум — не ломать чтение, но новые такие состояния не создавать.

Учти **`meritTransfer` / `eventPostId`**: правила из `business-merit-transfer.mdc` должны оставаться согласованными с «только участники комьюнити».

## 3. Критерии приёмки (дополнение к PRD)

- Не-участник комьюнити не попадает в RSVP ивента ни через основной флоу, ни через инвайт, пока не станет участником.
- После начала ивента запрос `unattend` от участника возвращает ошибку; UI не предлагает отмену.
- После админской отметки «пришёл» или «не пришёл» участник не может снять RSVP; «не пришёл» не удаляет строку участника из учёта ивента (остаётся в списке для админа/истории с явкой `no_show`).
- Автор и лид оба успешно вызывают `checkInByToken` и ручную установку явки.
