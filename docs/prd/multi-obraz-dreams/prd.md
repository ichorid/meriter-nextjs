Работа ведётся в **форке** и/или в **долгоживущей ветке форка**, синхронизируемой с `upstream` по необходимости.

## 1. Цель

Сделать **изолированный продуктовый пилот** для сценария **«Мультиобраз»**: пользователь с мобильного входа видит **простую витрину** (плашка + описание + один главный CTA), создаёт **«мечту»** с **минимальным набором полей**, после чего мечта **появляется в общей ленте пилота** и дальше живёт как **кооперативный проект** (`isProject`) в текущей модели данных, но **в UI остаются только задачи и обсуждения** и облегчённый контентный слой (без кошелька, биржи, долей, инвестиций, смены привязки к сообществу и прочего «тяжёлого» слоя). Обсуждения сделать **очевидными**: явная кнопка комментирования и **читаемый поток** комментариев без лишней когнитивной нагрузки.

**Зачем:** быстрый пилот на отдельном домене/аудитории без переписывания ядра Meriter и без риска сломать основной `dev`.

## 2. Контекст (как устроено сейчас)

- **ОБ / Future Visions:** хаб `typeTag === 'future-vision'`, лента карточек сообществ, маршрут вида `/meriter/future-visions` (см. `@.cursor/rules/business-future-vision.mdc`).
- **Кооперативные проекты:** `isProject` communities, задачи, тикеты/обсуждения, `CommunityWallet`, публикация на Биржу с сущности проекта — см. `@.cursor/rules/business-projects.mdc`, `@.cursor/rules/business-content.mdc`, `@.cursor/rules/business-investing.mdc`.
- **Создание проекта (web):** `web/src/app/meriter/projects/create/`, форма `CreateProjectForm` — имя, описание, выбор «родителя» / персональный проект, **доли основателя/инвестора**, инвестиции, рубрикатор **ценностей** (`ValuesFormPickerFields` / `futureVisionTags`), обложка для ветки «новое сообщество» и т.д.
- **Страница проекта:** множество вкладок и действий (кошелёк, команда, Биржа, инвестиции — зависит от настроек и флагов).
- **Аутентификация в продукте:** JWT/cookie, tRPC, возможны Telegram Mini App, fake auth и др. (см. `@.cursor/rules/index.mdc`). Пилот **обязан** явно выбрать поддерживаемые методы входа (раздел 18).

**Идея пилота:** не вводить новую сущность БД «мечта», а **переиспользовать проект** (`isProject`) + **отдельный UX-слой** и **узкий контракт создания**, плюс **фильтр/хаб** для ленты пилота.

## 3. Принципы и не-цели

### 3.1 Принципы

1. **KISS / один источник истины в данных:** «мечта» = проект с теми же API/коллекциями, что и сегодня, пока не доказана необходимость отдельного типа.
2. **Изоляция от `dev`:** код и документация пилота живут в **форке**; merge в upstream — только после явного gate (§11.3).
3. **Два уровня включения пилота:** (а) **деплой** на отдельном домене/стенде; (б) **фича-флаг** `PILOT_MODE` на web и api. Оба должны быть согласованы в runbook (§18).
4. **Defense in depth:** всё, что скрыто в UI пилота, **по возможности запрещено или игнорируется на сервере** для сущностей, помеченных как мечты пилота (`pilotContext` / флаг на community), чтобы не полагаться только на «безопасность через незнание URL».
5. **Обратная совместимость основного продукта** при выключенном `PILOT_MODE` на том же коде после merge: поведение Meriter **идентично** текущему (регрессионные тесты).
6. **Наблюдаемость:** прогресс-лог разработки (обязательно) + продуктовые события (желательно v1).

### 3.2 Не-цели (v1 пилота)

- Полноценная **Биржа**, рейтинг мечт, **инвестиции**, **доли основателя/инвестора**.
- **Кошелёк заслуг** и любые операции с балансом в UI пилота.
- **Смена привязки** мечты к другому сообществу / сложные админские действия в UI.
- **Голосовалки / опросы** внутри мечты (отдельный PRD при необходимости).
- **Полная копия** всех бизнес-правил Meriter в отдельной книге — только **дельта** для пилота (§10).

### 3.3 Глоссарий (канон для PRD и задач)

| Термин в PRD | Значение в системе | Примечание |
|--------------|-------------------|------------|
| **Мечта** | `Community` с `isProject === true` (и пометка пилота — см. TR) | В UI никогда не показывать пользователю слово «проект» в пилот-режиме, если продукт решит иначе — завести i18n override |
| **Лента пилота** | Список мечт, отфильтрованный правилом пилота | Не смешивать с глобальным `/meriter/projects` без явного решения |
| **Хаб пилота** | Логическое место «все мечты мультиобраза» | Может быть отдельный community-parent или тег — см. §14 рекомендации |
| **Обсуждение** | Тикет / тема форума в проекте (существующий модуль) | Термин в UI согласовать с копирайтом («Обсуждение» / «Тема») |
| **Участники** | Бывш. «Команда» | Только люди и роли; без кошелька и инвестиций в UI |
| **PILOT_MODE** | Совокупность env-флагов web+api | Имена переменных финализировать в первом коммите конфига |

## 4. Персоны и сценарии

### 4.1 Персоны

- **Участник пилота (мобильный):** заходит на домен пилота, не обязан понимать «проект vs сообщество».
- **Автор мечты:** создаёт мечту, дополняет контентом (текст/фото и т.д. по выбранному потоку §14).
- **Участник обсуждения:** читает темы, комментирует при наличии прав.
- **Исполнитель задач:** создаёт/обновляет/закрывает задачи (если правила сообщества разрешают).
- **Модератор / суперюзер (опционально v1):** скрытие мечты из ленты, блок — только если включено в scope (§14).

### 4.2 Главный user journey (E2E)

1. Пользователь открывает пилот-домен → **герой-блок «Мультиобраз»** + краткое описание.
2. Primary CTA → **мастер мечты**.
3. Заполняет **название** (обяз.) и **описание** (обяз. — см. AC-2a).
4. Нет полей: владелец, ценности, доли, инвестиции, Биржа, кошелёк.
5. Сабмит → мечта создана → **редирект** по политике §14.
6. Мечта в **ленте пилота**; переход внутрь → shell пилота: контент мечты + **задачи** + **обсуждения** + **участники** (урезанно).
7. Обсуждения: заголовок темы + **кнопка «Комментировать»** + чтение комментариев в выбранном паттерне (§5.5, §14).

### 4.3 Дополнительные сценарии (обязательно учесть при реализации)

| ID | Сценарий | Ожидание |
|----|----------|----------|
| **J-1** | Пользователь **уже залогинен** (cookie с основного домена недоступен) | Нормальный логин на домене пилота; после логина — возврат на intended URL мастера/мечты |
| **J-2** | Пользователь жмёт **«Назад»** из мастера | Возврат на витрину без потери черновика **опционально** v1 (если не делаем — явно «черновик не сохраняется») |
| **J-3** | **Deep link** на `/meriter/projects/[id]` с пилот-домена | Открывается **та же** мечта в **пилот-shell** (рекомендация по умолчанию), без обхода скрытых блоков через «полный» layout |
| **J-4** | Пользователь вручную подставил id **чужой** мечты / не пилот-проекта | Либо **404/доступ запрещён** по правилам community, либо показ в полном проекте **запрещён** в пилоте — **выбрать один** вариант в §14 и реализовать единообразно |
| **J-5** | API вернуло ошибку прав при комментарии | Понятное сообщение + **что сделать** (например «нужна роль X») без упоминания кошелька |
| **J-6** | Автор удалил аккаунт / заблокирован | Мечта остаётся или архивируется — **как в основном продукте**; в пилоте не менять модель без отдельного решения |

## 5. Функциональные требования

### 5.1 Витрина и навигация пилота

- [ ] **FR-1:** При `PILOT_MODE` стартовый экран показывает блок **«Мультиобраз»** + описание (2–4 предложения, согласование с продактом).
- [ ] **FR-2:** Одна **primary** CTA ведёт в **мастер мечты** (не создание сообщества, не полный enterprise-wizard).
- [ ] **FR-3:** Мобильная вёрстка — **первичный таргет** (360–430px); десктоп не ломается.
- [ ] **FR-4:** Глобальный сайдбар Meriter в пилот-layout **не рендерится** (или заменяется минимальной шапкой: логотип пилота, «Мои мечты»/лента, вход/профиль упрощённо — §5.7).
- [ ] **FR-25:** **FAB** и глобальные меню создания контента вне пилота **не показываются**, если они ведут на Биржу/кошелёк/инвестиции.

### 5.2 Создание «мечты»

- [ ] **FR-5:** Заголовки и сабмит — термины **«Мечта»** / **«Создать мечту»**; i18n группа `multiObraz.*` (или согласованное имя).
- [ ] **FR-6:** Поля v1: **название** (обяз.), **описание** (обяз. — см. **AC-2a**).
- [ ] **FR-7:** Нет UI выбора владельца / родительского сообщества; серверное поведение — §14.
- [ ] **FR-8:** Не отображать: доли, инвестиции, Биржа, кошелёк, ценности.
- [ ] **FR-9:** После успеха — редирект по §14.
- [ ] **FR-26:** Валидация на клиенте **и** на сервере: max length для названия/описания (конкретные числа взять из существующих лимитов community/project в коде при реализации и записать в прогресс-лог).

### 5.3 Лента / хаб пилота

- [ ] **FR-11:** Список мечт пилота с сортировкой **по дате создания убыв.** (v1; релевантность — backlog).
- [ ] **FR-12:** Элемент ленты ведёт на страницу мечты в **пилот-shell**.

### 5.4 Страница мечты (оболочка проекта)

- [ ] **FR-14:** Не рендерить: кошелёк, инвестиции, Биржа/опубликовать, доли, смена привязки к сообществу, merit transfer из участников, любые виджеты заслуг.
- [ ] **FR-15:** «Команда» → **«Участники»** (только `PILOT_MODE`).
- [ ] **FR-16:** **Задачи** и **обсуждения** — полный CRUD в рамках текущих прав сообщества (без новых типов сущностей).
- [ ] **FR-17:** Скрытия управляются **централизованно** (один хелпер `isPilotDreamShell` / `getPilotUiConfig`, без разрозненных проверок домена в 20 файлах).

- [ ] **FR-18:** У каждой темы в списке — **заметная** кнопка **«Комментировать»** (или «Ответить»).

### 5.7 Минимальная навигация пилота (чтобы не было «тупика»)

## 6. Технические требования

### 6.1 Режим пилота (включение)

- [ ] **TR-1:** Env-флаги: `NEXT_PUBLIC_PILOT_MODE` (web) и `PILOT_MODE` (api) — имена финализировать; значения `true`/`1`.
- [ ] **TR-2:** При `PILOT_MODE=false` на деплое: пилот-роуты **404** или редирект на обычный landing (единая политика на стенде); **никаких** пилот-only компонентов в SSR без проверки.

### 6.2 Идентификация «мечты пилота» на сервере

- [ ] **TR-4:** При создании передавать **`pilotContext: 'multi-obraz'`** (или константа в shared-types в форке) в существующую мутацию создания проекта.
- [ ] **TR-5:** Сервер помечает созданную community одним из способов (**выбрать один при реализации и задокументировать**):  
  - **A)** `settings` / произвольное поле `pilotMeta: { kind: 'multi-obraz' }` без миграции, если схема позволяет;  
  - **B)** `parentCommunityId === PILOT_HUB_COMMUNITY_ID` из env;  
  - **C)** отдельная коллекция маппинга `pilot_dreams` (тяжелее, не рекомендуется v1).  
  Рекомендация PRD: **B + опционально A** для фильтра ленты и defense-in-depth проверок.
- [ ] **TR-6:** Все **pilot-only** процедуры (если введены) проверяют флаг **`PILOT_MODE`** на api; при false — `FORBIDDEN` или `NOT_FOUND`.

### 6.3 Лента пилота

- [ ] **TR-8:** Без деградации перформанса относительно соседних лент (батчи, лимиты).

### 6.4 Запрет «опасных» мутаций для мечты пилота

Список **закрыть в `business-multi-obraz-pilot.mdc`** и дублировать в коде константой `PILOT_FORBIDDEN_MUTATIONS`.

## 7. Детали реализации по слоям

### 7.1 Backend (форк)

- Ветвления по `pilotContext` + пометка community.
- Seed: **один** `PILOT_HUB_COMMUNITY_ID` в конфиге стенда; инструкция в `SETUP-FORK.md`. Для прод-пилота — runbook создания hub вручную + env.

## 14. Зафиксированные решения по умолчанию (рекомендации PRD для старта реализации)

Чтобы не блокировать разработку, ниже **дефолты**. Продукт может переопределить до `Approved`, иначе команда идёт по ним.

| Тема | Решение по умолчанию |
|------|----------------------|
| **Вход** | Только **авторизованные** пользователи создают мечты; витрина и лента **публично читаемы** только если текущая модель community **public** — иначе лента только для участников (как сейчас). Кнопка CTA для гостя → экран логина → возврат. |
| **Host vs path** | Prod: **по Host** (`cw.ru` = пилот layout). Dev: **path** `/pilot/multi-obraz` + `PILOT_MODE=true`. |
| **Привязка мечты** | Создание как **дочерний проект** хаба: `parentCommunityId = PILOT_HUB_COMMUNITY_ID` (env); **не** персональный проект — чтобы все мечты были в одном контексте ленты и модерации. |
| **Персональный «только автор»** | Достигается UX (не показываем выбор parent) + роль автора как lead; **не** отдельная сущность. |
| **Редирект после create** | На **страницу созданной мечты** (удобнее продолжить контент); вторичная ссылка «К ленте» в шапке. |
| **Пост с камнем/фото** | После входа на страницу мечты показать **одинразовый** inline prompt / баннер «Расскажите о мечте» с кнопкой → существующий flow создания **публикации** в community мечты (тип поста — по умолчанию `basic`, не event). |
| **Паттерн обсуждений** | Список тем; **аккордеон** с комментариями + кнопка «Комментировать»; опциональная отдельная URL для шаринга темы. |
| **Участники** | Показать список; **скрыть** все действия с заслугами и инвайтами, ведущие в кошелёк; если без них список пустой — показать текст «Приглашения позже» или только аватар автора — согласовать копирайт. |
| **Deep link не пилот-проекта** | **403/404** в пилот-shell (не подмешивать полный Meriter), чтобы не обходить матрицу §10. |
| **Уведомления** | **Скрыть** центр уведомлений v1. |
| **Локаль** | **ru** по умолчанию на пилот-домене. |

*Противоречие с ранним черновиком «только personalProject»:* для ленты и модерации дефолт изменён на **hub child**. Если продукт настаивает на personal — лента должна фильтровать по `pilotMeta`/флагу (тогда нужен вариант **A** в TR-5).
