# Развёртывание «Мультиобраз» (пилот) на сервере

Кратко для DevOps / коллеги: это **тот же репозиторий и те же образы**, что и у Meriter, но с **включёнными флагами пилота** и **одним заранее созданным hub-сообществом** в MongoDB. Отличия только в **переменных окружения** и в **ветке** (если у вас принято: `pilot` / `feat/multi-obraz-pilot`).

Подробный runbook по переменным и smoke: `SETUP-FORK.md` (в этой же папке).

---

## Чем пилот отличается от обычного Meriter

| Область | Обычный Meriter | Пилот «Мультиобраз» |
|--------|-----------------|---------------------|
| **Web** | Главная логика на `/meriter/...` | При `NEXT_PUBLIC_PILOT_MODE=true`: витрина на **`/`**, мастер мечты **`/create`**, профиль пилота **`/profile`**; часть UI мечты без ухода в «полный» Meriter где возможно |
| **API** | `PILOT_MODE` выкл. | `PILOT_MODE=true`: разрешён `project.create` с `pilotContext: 'multi-obraz'`, лента `pilotDreamFeed`, часть мутаций для мечты заблокирована (TR-14) |
| **Данные** | Любые сообщества/проекты по правилам продукта | Нужен **один** `team`-хаб: его `id` в `PILOT_HUB_COMMUNITY_ID` / `NEXT_PUBLIC_PILOT_HUB_COMMUNITY_ID` — родитель для мечт |
| **Сборка web** | Без пилотских build-args | В Docker: `NEXT_PUBLIC_PILOT_*` передаются **на этапе build** образа `web` — после смены флагов нужен **`docker compose build --no-cache web`** (или `up --build`) |

Итого на сервере: **те же** Mongo, S3, JWT, OAuth и т.д., что и для Meriter; **дополнительно** задаёте пилот-флаги и id хаба.

## Шаги на удалённом сервере

1. **Сборка из нужной ветки**  
   Задеплойте образы из ветки **`pilot`** или **`feat/multi-obraz-pilot`** (или merge в вашу релизную ветку — политика команды).

2. **Mongo**  
   Убедитесь, что БД такая же по версии/реплике, как для Meriter (см. основной `docker-compose.yml` / прод-гайд).

3. **Создать hub**  
   Один раз в этой БД нужно **командное сообщество** с допустимым для родителя проекта типом (обычно `typeTag: 'team'`), не глобальный хаб типа marathon-of-good.  
   Скопировать его **`id`** в переменные ниже.

4. **Переменные окружения** (значения примерные):

   **API / общий `.env`:**
   - `PILOT_MODE=true`
   - `PILOT_HUB_COMMUNITY_ID=<ObjectId хаба>`

   **Web (и для Next build-args в Docker):**
   - `NEXT_PUBLIC_PILOT_MODE=true`
   - `NEXT_PUBLIC_PILOT_HUB_COMMUNITY_ID=<тот же ObjectId>`
   - Опционально: `NEXT_PUBLIC_PILOT_STANDALONE=true` — скрыть dev-chrome на демо-стенде

   **Важно:** id на web и api **должен совпадать**. Без hub API вернёт ошибку при создании мечты.

5. **Пересборка web**  
   После смены `NEXT_PUBLIC_*` обязательно **пересобрать** образ фронта (иначе клиент останется со старыми константами).

6. **Проверка**  
   - С пилотом: открыть `/`, `/create`, создать мечту, открыть `/meriter/projects/{id}`.  
   - Выключить пилот (`false` / убрать): поведение как у обычного Meriter; `pilotDreamFeed` и пилот-create дают 403 на API.

## Локально на Windows (одна команда по ветке)

Из корня репозитория:

```powershell
.\scripts\windows\run-docker-local.ps1
```

- Ветка **`dev`**: скрипт **принудительно выключает** пилот для этой сессии `docker compose` (даже если в `.env` остались старые `true`).
- Ветки **`pilot`** или **`feat/multi-obraz-pilot`**: скрипт **включает** пилот; id хаба берите из корневого `.env`.
- Остальные ветки: скрипт **не трогает** пилот-переменные — используется только `.env`.

Для фона: `docker-compose.local.yml` прокидывает в контейнер **api** переменные `PILOT_MODE` и `PILOT_HUB_COMMUNITY_ID`.
