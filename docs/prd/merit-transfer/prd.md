# PRD: Передача заслуг (Merit Transfer)

## Цель
Дать участникам возможность напрямую передавать заслуги друг другу с обязательным комментарием, создавая публичный лог благодарностей внутри сообщества и в профиле.

## Контекст
- **Текущее состояние**: Начислить мериты другому можно только голосованием за пост или (админ) эмиссией из воздуха. Прямой передачи нет. Публичного лога нет.
- **Проблема**: Нет способа сказать «спасибо» конкретному человеку и подкрепить это заслугами. Нет прозрачности — кто кому что начислил.
- **Связанные модули**: Wallets (merit-resolver), Community members list, User profile, Comments (для автокомментариев в ивентах).
- **Зависимость**: Фича «Ивенты» использует этот механизм для начисления заслуг участникам ивента.
- **Бизнес-документ**: `docs/specs/Meriter_Передача_заслуг_v1.md`

## Требования

### Функциональные
- [ ] FR-1: Кнопка «Передать заслуги» у каждого участника в списке участников сообщества и проекта. Видна всем участникам, **кроме собственной строки** (себе передать нельзя).
- [ ] FR-2: Диалог передачи: получатель (предзаполнен), количество, обязательный комментарий, выбор кошелька-источника, выбор кошелька-назначения (если источник — глобальный). **Квота исключена из источников — только wallet** (локальный или глобальный).
- [ ] FR-3: Маршрутизация: локальные заслуги → только на тот же локальный кошелёк. Глобальные → на выбор: глобальный получателя или локальный (конвертация 1:1).
- [ ] FR-4: Валидация: количество > 0, достаточно средств, `senderId ≠ receiverId`, получатель — участник того же контекста (для локальных). Комментарий обязателен **только на клиенте** (UI не даёт отправить без него). На уровне БД и API поле `comment` — опциональное (так гибче для будущих сценариев).
- [ ] FR-5: Раздел «Переданные заслуги» в сообществе и проекте — лента записей: кто, кому, сколько, откуда→куда, за что, когда. Новые сверху. Видна всем участникам.
- [ ] FR-6: Раздел «Заслуги» в профиле пользователя — две вкладки: «Входящие» и «Исходящие». Та же информация.
- [ ] FR-7: Существующая админская кнопка «Начислить заслуги» (эмиссия) остаётся без изменений.

### Технические
- [ ] TR-1: Новая модель `MeritTransfer` в MongoDB (отдельная коллекция). Поле `comment` — **опциональное** на уровне схемы и API (обязательность обеспечивается клиентом).
- [ ] TR-2: Новый сервис `MeritTransferService` — валидация, дебет/кредит через существующий merit-resolver, создание записи.
- [ ] TR-3: Новый tRPC роутер `meritTransfer` — `create`, `getByCommunity`, `getByUser`.
- [ ] TR-4: Permission check в роутере: отправитель — участник сообщества/проекта.
- [ ] TR-5: UI-компоненты: кнопка, диалог, лента записей, вкладки профиля.
- [ ] TR-6: Реиспользуемый механизм — чтобы ивенты вызывали тот же сервис + дополнительно создавали автокомментарий.

### Cursor должен проверить в коде
- [ ] CHECK-1: Как устроен merit-resolver: найти точное место, где wallet дебетуется при голосовании. Понять разницу между quota и wallet debit. Убедиться, что transfer использует wallet-only path.
- [ ] CHECK-2: Как устроен список участников (MembersList или аналог): где рендерится, какие данные получает, как добавить кнопку рядом с каждым участником.
- [ ] CHECK-3: Как устроена админская «Начислить заслуги» (эмиссия): найти endpoint и UI. Убедиться, что новая кнопка «Передать заслуги» не конфликтует и не ломает существующую.
- [ ] CHECK-4: Профиль пользователя: какие секции/табы уже есть, как добавить новую вкладку «Заслуги».
- [ ] CHECK-5: Навигация внутри сообщества/проекта: как добавить раздел «Переданные заслуги» (паттерн из существующих разделов).

## Детали реализации

### Backend
- **Новый сервис**: `merit-transfer.service.ts` — create, getByContext, getByUser
- **Новый роутер**: `merit-transfer.router.ts` — create (protected, member check), queries
- **Схема**: `MeritTransfer` — senderId, receiverId, amount, comment?, sourceWalletType, sourceContextId, targetWalletType, targetContextId, communityContextId, eventPostId?, createdAt. **Важно**: `comment` опционален на уровне схемы/API, обязательность — только на клиенте.
- **Изменения в существующих**: Нет. Используем существующий merit-resolver для дебет/кредит операций.

### Frontend
- **Новый компонент**: `MeritTransferButton` — кнопка у участника
- **Новый компонент**: `MeritTransferDialog` — форма передачи
- **Новый компонент**: `MeritTransferFeed` — лента записей (реиспользуется в сообществе, проекте, профиле)
- **Изменения**: Список участников сообщества/проекта — добавить `MeritTransferButton`
- **Изменения**: Профиль пользователя — добавить вкладку «Заслуги»
- **Изменения**: Страница сообщества/проекта — добавить раздел «Переданные заслуги»

## Acceptance Criteria
- [ ] AC-1: Участник сообщества видит кнопку «Передать заслуги» у других участников в списке, но НЕ у себя
- [ ] AC-2: Диалог открывается, валидирует ввод, выполняет перевод, баланс отправителя уменьшается, получателя — увеличивается
- [ ] AC-2a: В диалоге доступны только wallet-кошельки (квота отсутствует в выборе источника)
- [ ] AC-2b: API отклоняет передачу себе (`senderId = receiverId`)
- [ ] AC-3: Запись появляется в «Переданные заслуги» сообщества сразу после передачи
- [ ] AC-4: Запись появляется в профиле отправителя (исходящие) и получателя (входящие)
- [ ] AC-5: Админская «Начислить заслуги» работает как раньше
- [ ] AC-6: Передача из глобального кошелька на локальный конвертирует 1:1
- [ ] AC-7: `pnpm lint && pnpm build` проходят без ошибок

## Критические тесты
- [ ] **MeritTransferService.create**: передача 50 из локального wallet (balance=100) → sender.balance=50, receiver.balance+=50, MeritTransfer запись создана.
- [ ] **MeritTransferService.create**: передача при недостаточном балансе → ошибка, балансы не изменились, MeritTransfer запись НЕ создана.
- [ ] **MeritTransferService.create**: senderId = receiverId → ошибка 400.
- [ ] **MeritTransferService.create**: глобальный wallet → локальный кошелёк получателя → конвертация 1:1, оба баланса корректны.
- [ ] **MeritTransferService.create**: передача с eventPostId → запись содержит eventPostId (для будущего автокомментария).
- [ ] **Роутер**: не-участник сообщества пытается передать → ошибка доступа.
- [ ] **Существующая эмиссия**: после реализации transfer — админская «Начислить заслуги» по-прежнему работает без изменений.

## Заметки
- Этот механизм — зависимость для фичи «Ивенты». Ивенты добавят кнопку в RSVP-список + автокомментарий, используя тот же сервис.
- Квота НЕ может использоваться для передачи (только wallet).
- На будущее: возможно потребуется лимит на передачи (антиспам). Пока не реализуем.
- **Краевой случай (ивенты)**: приглашённые по инвайт-ссылке пользователи попадают в `eventAttendees`, но НЕ являются участниками сообщества/проекта. Им можно начислить заслуги только из глобального кошелька → на их глобальный кошелёк. Локальная передача для них невозможна (нет локального контекста).
