## 1. Цель

Сократить вертикальный «шум» на странице **сообщества** и **кооперативного проекта** (`isProject`): убрать громоздкие карточки-ссылки на дочерние экраны, перенести **историю заслуг** в осмысленное место с расширенным содержанием, а **проекты сообщества, ивенты и посты на Бирже** встроить в **единую ленту** с вкладками и контекстными фильтрами/действиями над блоком управления лентой (как сейчас серый блок «Создать пост / поиск / сорт / Фильтры»).

**Успех (измеримо):**

- На типичном десктопе блок «дашборд + переходы» не занимает больше **одного экрана** до начала ленты (для сценария Birzha-source community с кошельком и участниками).
- Пользователь **не покидает** основную страницу сообщества/проекта для просмотра списков проектов, ивентов и постов на Бирже (кроме перехода в **детальную карточку** сущности).
- Участник видит **«История заслуг»** рядом с участниками; внутри — **полная** картина движений заслуг в контексте общины/проекта (см. §6), а не только peer-передачи.

## 2. Область применения (in scope)

### 2.1. Поверхности

| Поверхность | Файлы-ориентиры (текущий код) |
|-------------|-------------------------------|
| Страница сообщества | `web/src/app/meriter/communities/[id]/CommunityPageClient.tsx`, `CommunityHeroCard`, `CommunityDashboard`, `BirzhaSourcePostsEntryRow` |
| Страница кооперативного проекта | `web/src/app/meriter/projects/[id]/ProjectPageClient.tsx`, `ProjectDashboard`, `ProjectWorkArea`, `BirzhaSourcePostsEntryRow` |
| Дочерние маршруты сообщества | `.../events`, `.../projects`, `.../birzha-posts`, `.../merit-transfers` (страницы под `web/src/app/meriter/communities/[id]/`) |
| Дочерние маршруты проекта | `.../events`, `.../birzha-posts`, `.../merit-transfers` (под `projects/[id]/`) |
| Навигация / константы | `web/src/lib/constants/routes.ts`, `web/src/lib/routing/route-patterns.ts`, тесты паттернов |
| i18n | `web/messages/*.json` (ключи `pages.communities`, `meritTransfer`, `projects`, и др.) |

### 2.2. Вне области (out of scope) — явно

- Редизайн **глобальной** ленты МД (`marathon-of-good`) как отдельного продукта, **кроме** случаев, когда на странице МД переиспользуется тот же компонент вкладок (если команда решит унифицировать — отдельное решение в фазе согласования).
- Изменение **бизнес-правил** начисления/списания заслуг (только отображение и навигация).
- Мобильная **нижняя навигация** Meriter (5 вкладок) — без изменения маршрутов табов, если не согласовано отдельно.

## 3. Термины (продукт ↔ код)

| Продукт (RU) | Смысл |
|----------------|--------|
| **История заслуг** (в общине) | Лента операций/событий, связанных с заслугами **в контексте** данной общины или проекта: кто, что, когда, сколько (не ограничиваясь P2P). |
| **Прямые передачи** (legacy) | Текущий `meritTransfer.getByCommunity` на странице `.../merit-transfers`. |
| **Вкладка «Посты»** | Текущая взвешенная лента публикаций/опросов общины (как сейчас под панелью управления). |
| **Вкладка «Проекты»** | Список дочерних кооперативных проектов, привязанных к сообществу-родителю (сейчас `CommunityProjectsPageClient`). |
| **Вкладка «Ивенты»** | Список публикаций `postType === 'event'` / данные `events.*` в UX, эквивалент текущего списка на `.../events`. |
| **Вкладка «Мы на бирже»** | Посты с `sourceEntity*` в МД (текущий `CommunityBirzhaPostsPageClient` / аналог для проекта). |

В пользовательском тексте по глоссарию платформы использовать **«Заслуги»**, не «мериты».

## 6. Целевое поведение: «История заслуг» в общине/проекте

### 6.1. Продуктовое определение

**FR-HIST-1:** В карточке **«Участники»** (компонент `CommunityDashboard`, правая колонка сетки) для **участников** общины отображается действие **«История заслуг»** (новый i18n-ключ), ведущее на маршрут детального просмотра истории **в контексте этой общины**.

**FR-HIST-2:** Содержимое экрана истории — **не** узкий список только `merit_transfers`, а **агрегированная лента** всех релевантных движений/событий заслуг, произошедших **в рамках данной общины или проекта**, с полями: **кто** (участник), **что за действие**, **когда**, **сколько заслуг** (знак приход/расход по согласованной семантике), вторичные ссылки на пост/ивент/передачу там, где возможно безопасно обогатить (по аналогии с `merit-history-enrichment`).

**FR-HIST-3:** Для **гостя и не-участника** ссылка в карточке участников **не отображается** (как сейчас скрывается строка peer-передач для не-участников).

**FR-HIST-4:** Заголовок страницы и описание — **«История заслуг»** + короткое пояснение (не «Прямые передачи заслуг в сообществе»).

### 6.2. Техническая проработка (обязательный spike до кодирования UI)

**TR-HIST-0 (spike):** Зафиксировать **источник истины** для «всех движений в общине»:

- Варианты: (A) выборка из **`transactions`** по критерию «кошелёк с `communityId` = этой общине» для **всех** участников; (B) объединение wallet-строк с фильтром по **обогащённому** контексту публикации/ивента; (C) отдельная проекция/коллекция (обычно нежелательно на MVP).
- Требования к ответу API: пагинация (cursor или page), сортировка по времени, **категории-фильтры** в духе профильной истории (хотя бы: передачи, голоса/комментарии, инвестиции, тапалка, системные, «все»).
- **Приватность:** история в общине видна только **участникам** этой общины (и суперадмин); не раскрывать данные не-участникам; не показывать лишнее для удалённых пользователей — уточнить при моделировании.

**TR-HIST-1:** Новая процедура tRPC (имя на усмотрение реализации, например `wallets.getCommunityMeritHistory` **или** `communities.getMeritHistory`) с Zod-input: `communityId` / `projectId`, пагинация, фильтры категорий, опционально период для KPI, если дашборд включается в ту же страницу.

**TR-HIST-2:** Повторное использование **`merit-history-enrichment`** / общих мапперов из `@docs/prd/merit-history` где возможно; избежать N+1.

**TR-HIST-3:** Маршрутизация web: предпочтительно **`/meriter/communities/[id]/merit-history`** и **`/meriter/projects/[id]/merit-history`**; с **`/merit-transfers`** — **301/302 редирект** на новый URL (сохранить закладки и внешние ссылки).

**TR-HIST-4:** Обновить **`business-merit-history.mdc`**: таблица маршрутов и процедур для нового потока; явно разделить **peer-only** (`getByCommunity` для узких кейсов, если останется) и **полная история общины**.

## 7. Целевое поведение: вкладки ленты

### 7.1. Размещение

**FR-TAB-1:** Непосредственно **над** существующим блоком панели ленты (`rounded-xl bg-gray-100 dark:bg-gray-800/50 p-5 ... space-y-4`) разместить **горизонтальный переключатель вкладок** из четырёх пунктов:

1. **Посты**  
2. **Проекты**  
3. **Ивенты**  
4. **Мы на бирже**

**FR-TAB-2:** Визуальный стиль — в духе **Obsidian Nocturne** (`design-system.mdc`): слои без лишних бордеров, активная вкладка читаема в dark/light, контраст фокуса для клавиатуры.

**FR-TAB-3:** Состояние активной вкладки хранить в **URL** (рекомендуется `?feedTab=posts|projects|events|birzha` или эквивалент), чтобы поддержать шаринг и кнопку «назад» браузера. Сохранить совместимость с существующими query: `post`, `poll`, `q`, `sort`, `vt`, … — не ломать диплинки.

### 7.2. Проект (`isProject`)

**FR-TAB-PROJ-1:** Для страницы проекта внедрить **ту же модель**, где применимо: **Посты** (работа/тикеты — уточнить, не дублировать `ProjectWorkArea`), **Ивенты**, **Мы на бирже**; вкладка **«Проекты сообщества»** на странице **проекта** не показывается (N/A). Если «Посты» на проекте семантически совпадают с `ProjectWorkArea`, в PRD реализации зафиксировать **один** канонический список (вкладка vs блок ниже) — избежать дублирования контента.

**FR-TAB-PROJ-2:** **«История заслуг»** для участника проекта — в **`ProjectDashboard`** рядом с метриками участников / в согласованном слоте (зеркало `CommunityDashboard`).

### 7.3. Сообщества без Birzha-source дашборда

Для ветки `else` в `CommunityPageClient` (сетка «Участники», МД-only и т.д.) — **минимум**:

**FR-TAB-EDGE-1:** Если у общины есть **ивенты** и/или **проекты** в текущем UX, перенести их в ту же модель вкладок **или** явно описать в реализации «фаза 2» с обоснованием. PRD требует **единого** паттерна для всех локальных общин, где уже есть лента + вторичные списки, чтобы не поддерживать два дизайна вечно.

## 8. Панель действий и фильтров под вкладкой

Общий контейнер — тот же `rounded-xl ... space-y-4`, наполнение **зависит от вкладки**.

### 8.1. Вкладка «Посты»

**FR-POST-1:** Сохранить текущее поведение: **Создать пост** (если доступно), иконки **поиск / недавние / тренд** (или текущие эквиваленты), **Фильтры** (таксономия, теги, категории — как сейчас).

### 8.2. Вкладка «Проекты»

**FR-PROJ-1:** Перенести кнопку **«Создать проект»** внутрь панели (как primary action слева или по паттерну primary/secondary дизайн-системы).

**FR-PROJ-2:** **Поиск** — поле поиска по названию/описанию дочерних проектов (если API отсутствует — client-side filter по загруженной странице + задача на API в бэклоге).

**FR-PROJ-3:** **Фильтры** (предложение по умолчанию):

| Фильтр | Значения |
|--------|----------|
| Статус жизненного цикла | **Активные** (по умолчанию) / **Архивные** / **Закрытые** / **Все** |
| Сортировка | По дате обновления / по названию |

Уточнить по фактическим полям `projectStatus` и спискам в `useGlobalProjectsList` / `communities.*`.

### 8.3. Вкладка «Ивенты»

**FR-EVT-1:** Перенести **«Новый ивент»** в панель (доступность по `settings.eventCreation` и ролям — как на текущей странице ивентов).

**FR-EVT-2:** **Поиск** по заголовку/описанию (client или server — по возможностям текущего `events.*`).

**FR-EVT-3:** **Фильтры времени:**

| Фильтр | Значения |
|--------|----------|
| Окно времени | **Предстоящие и текущие** / **Прошедшие** / **Все** |

Отображение секций как на текущем экране — **либо** две под-вкладки, **либо** один список с группировкой (решение UX в реализации; поведение списка не хуже текущего).

### 8.4. Вкладка «Мы на бирже»

**FR-BIRZHA-1:** Перенести **«Создать пост на бирже»** / `PublishToBirzhaButton` в панель по тем же правам, что сейчас на `BirzhaSourcePostsEntryRow`.

**FR-BIRZHA-2:** **Поиск** по тексту поста (если есть API; иначе — фильтр на загруженном наборе с лимитом и дисклеймером).

**FR-BIRZHA-3:** **Фильтры** (предложение):

| Фильтр | Значения |
|--------|----------|
| Статус публикации | **Активные** / **Закрытые** / **Все** |
| Сортировка | По дате публикации на Бирже / по рейтингу (если данные доступны без тяжёлого запроса) |

Согласовать с `CommunityBirzhaPostsPageClient` и процедурами выборки.

### 8.5. Общие требования к панели

**FR-PANEL-1:** Одна колонка на мобиле; на `sm+` — компактный ряд (primary action + search + filters), без вертикальной «простыни» из трёх больших карточек.

**FR-PANEL-2:** Сохранить **telemetry** (`trackMeriterUiEvent`) для создания поста/проекта/ивента/биржи — обновить `surface`/`tab` в payload.

**FR-CONTENT-1:** Под панелью рендерить **тот же тип списка**, что сейчас на дочерней странице: карточки проектов, карточки ивентов, карточки постов Биржи, лента публикаций — с **переиспользованием** существующих компонентов карточек где возможно.

**FR-CONTENT-3:** Пустые состояния и скелетоны — для каждой вкладки отдельно, копирайт на русском по глоссарию.

- Тонкая обёртка, которая **сразу** редиректит (Server Component + `redirect()`).

**FR-MIG-2:** Обновить все **внутренние ссылки** в приложении (sidebar, уведомления, deep links из `routes.*`, `CreateMenu`, и т.д.) на новый формат URL.

**FR-MIG-3:** Обновить **`route-patterns.ts`** и **`route-patterns.test.ts`**: статические/dynamic паттерны, sitemap при наличии.

**FR-MIG-4:** Документировать для поддержки: старые URL остаются валидными через редирект минимум **2 релиза** (политика на усмотрение, зафиксировать в changelog).

## 13. Тестирование

### 13.1. API

- [ ] Участник общины получает непустую историю при наличии транзакций в БД в scope общины.
