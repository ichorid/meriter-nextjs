## 1. Контекст и бизнес-потребность

### 1.1 Что есть сейчас
- В каждом **сообществе** есть текст «Образ Будущего» (ОБ) — декларация ценностей сообщества. Сейчас это простое текстовое поле, доступное на редактирование только админу сообщества.
- В каждом **проекте** (= community с `typeTag='team'`) есть «Описание проекта». Сейчас это простое текстовое поле, доступное на редактирование только админу проекта.
- В ленте сообщества вкладки: **Посты / Проекты / Ивенты / Мы на бирже**.
- В ленте проекта вкладки: **Посты / Ивенты / Мы на бирже**.

### 1.2 Что хотим
1. Превратить ОБ и Описание из «одиночного admin-only текстового поля» в **коллективно согласуемый текст** (совместный документ):
   - есть **официальный текст** (то, что вошло в документ);
   - участники могут **предлагать варианты** формулировок отдельных блоков;
   - сообщество **голосует заслугами** за варианты;
   - **админ** может закрепить формулировку волевым решением;
   - к каждому варианту можно прикрепить **ссылки-обоснования**.
2. Заложить **архитектурную возможность** создавать любые другие совместные документы в сообществе/проекте, помимо ОБ/Описания. На релизе эта возможность по умолчанию выключена (но реализована).
3. Добавить **простой WYSIWYG-редактор** для текстовых полей (дочерняя фича — см. §22).

### 1.3 Принципы UX
- **Прозрачность**: всегда видно, почему текст стал официальным — по голосованию или волевым решением.
- **Не «вики»**: документ — это одна простыня с секциями/блоками, без перекрёстных ссылок и истории отдельных «страниц».
- **Один источник правды**: для ОБ — один документ на сообщество. Для Описания — один документ на проект.

## 2. Скоуп: что шипим vs архитектурная готовность

### 2.1 Скоуп релиза (обязательно в проде)
Реализуется и работает «из коробки»:
- Новые настройки сообщества/проекта (см. §5): `documentsMode`, `documentCreators`, экономические настройки документов.
- **По умолчанию** для всех **существующих и новых** сообществ и проектов `documentsMode = 'visionOrDescriptionOnly'` (то есть совместный режим включён для ОБ/Описания, но не для кастомных документов). См. §5.1 про миграцию.
- Документ-обёртка над существующим текстом ОБ (для сообществ) и Описания (для проектов): структура секций/блоков, варианты, голосование, применение победителя, антиспам.
- Вся механика, описанная в §10–§17 (флоу варианта, голосование, применение, закрытие, ссылки-обоснования, история).
- Минимальный WYSIWYG (§22).

### 2.2 Архитектурная готовность (реализовано, но выключено по умолчанию)
Должно быть полностью реализовано в коде, доступно через настройку `documentsMode='all'`:
- Вкладка **«Документы»** в ленте сообщества и проекта (см. §6 — UI и мобильная навигация).
- Возможность создавать **произвольные** совместные документы (помимо ОБ/Описания) — см. §11.
- Настройка `documentCreators` (кто может создавать кастомные документы: admins | members).

> Бизнес-смысл: на старте включаем «Только для ОБ/Описания» во всех сообществах, чтобы получить ценность фичи без рисков спама. Когда поймём, что фича стабильна и понятна — переключаем выбранные сообщества на `'all'`.

### 2.3 Что НЕ входит в скоуп
- Полноценная вики / иерархия документов / древовидные ссылки.
- Совместное редактирование в реальном времени (одновременная правка курсора, OT/CRDT) — НЕ нужно. Работаем по модели «вариант → голосование → применение».
- Версионная история с возможностью отката (можно отдельной фичей в v2.0). Минимальная история **есть** (§19), но без UI отката.
- Импорт из Google Docs (см. §22.3).
- Изображения и код-блоки в WYSIWYG (см. §22.3).
- Уведомления о новых вариантах/применениях — в текущем скоупе используем существующую систему нотификаций (см. §20.7), без новых типов нотификаций сверх необходимого минимума.

## 3. Терминология

| Термин | Определение |
|---|---|
| **Документ** | Совместно редактируемый текстовый объект. Принадлежит сообществу или проекту. Имеет `type` (см. §4.1). |
| **ОБ-документ** | Документ типа `imageOfFuture`. Один на сообщество. Привязан к существующему полю ОБ сообщества. |
| **Описание-документ** | Документ типа `description`. Один на проект (community с `typeTag='team'`). Привязан к существующему полю «Описание проекта». |
| **Кастомный документ** | Документ типа `custom`. Любое количество в сообществе/проекте. Доступен только при `documentsMode='all'`. |
| **Секция** | Логический раздел документа (например, «Цели», «Принципы»). У документа всегда ≥1 секция. |
| **Блок** | Атом контента внутри секции: абзац, заголовок, маркированный/нумерованный список, цитата. У секции всегда ≥1 блок. |
| **Официальный текст блока** | Текущая принятая формулировка. То, что видит читатель. |
| **Вариант блока** | Предложенная альтернатива официальному тексту. Может иметь ссылки-обоснования. Участвует в голосовании. |
| **Голосование за вариант** | Заслуги (мериты), накопленные на варианте через голоса участников (+ и −). |
| **Победитель голосования** | Вариант с наибольшим рейтингом по завершении голосования. |
| **Применение варианта** | Действие, после которого текст варианта становится официальным текстом блока. |
| **Применение волевым решением** | Применение варианта (или произвольной формулировки) автором/админом без оглядки на результат голосования. |
| **Цикл голосования** | Период от появления первого варианта по блоку до закрытия голосования (см. §13). |

## 4. Сущности и data model

> Имена коллекций/полей — предложение. Курсор подстраивает под `domain/models/`, `libs/shared-types/`. Все схемы — Mongoose + Zod (см. `architecture.mdc`).

### 4.1 Document
Хранится в коллекции `documents` (или эквивалентной по соглашению кодовой базы).

| Поле | Тип | Описание |
|---|---|---|
| `_id` | ObjectId | |
| `communityId` | ObjectId | К какому сообществу/проекту принадлежит. Required. |
| `type` | enum: `'imageOfFuture' \| 'description' \| 'custom'` | Тип документа. |
| `title` | string | Для `type='custom'` — пользовательский. Для ОБ/Описания — фиксированный («Образ Будущего» / «Описание проекта»), не редактируется. |
| `sections` | Section[] | Список секций (см. §4.2). Минимум 1. |
| `mode` | enum: `'manual' \| 'auto'` | Режим применения победителя (§12). Default: `'manual'`. |
| `votingDurationHours` | number | Длительность цикла голосования (§13). Default: `48`. |
| `variantCost` | number | Стоимость предложить вариант (§14). Default: наследуется из `settings.postCost` сообщества. |
| `allowDownvotes` | boolean | Разрешить минусование вариантов (§15). Default: `true`. |
| `createdAt`, `updatedAt` | Date | |
| `createdBy` | UserId | Кто создал документ. Для ОБ/Описания — кто впервые перевёл текст в режим документа (или системный bootstrap-юзер, см. §5.1). |
| `status` | enum: `'active' \| 'archived'` | Архивные документы скрыты из UI. По умолчанию `'active'`. |
| `deleted`, `deletedAt` | boolean / Date | Soft delete по аналогии с публикациями. |

**Уникальные ограничения:**
- На сообщество может быть максимум **один** документ с `type='imageOfFuture'`.
- На проект (community с `typeTag='team'`) может быть максимум **один** документ с `type='description'`.
- Документы с `type='custom'` — без ограничений по количеству.

### 4.2 Section (вложенная в Document)

| Поле | Тип | Описание |
|---|---|---|
| `id` | string (uuid) | Стабильный id секции. |
| `title` | string | Заголовок секции (опционально, может быть пустым). |
| `order` | number | Порядковый номер. |
| `blocks` | Block[] | Список блоков. Минимум 1. |

### 4.3 Block (вложенный в Section)

| Поле | Тип | Описание |
|---|---|---|
| `id` | string (uuid) | Стабильный id блока. Не меняется при правках. |
| `order` | number | Порядковый номер внутри секции. |
| `blockType` | enum: `'paragraph' \| 'heading' \| 'list-bullet' \| 'list-numbered' \| 'quote'` | Тип блока. |
| `officialContent` | string (rich text, см. §22.2) | Текущая официальная формулировка. |
| `officialContentSetAt` | Date | Когда текст стал официальным. |
| `officialContentSetBy` | UserId | Кто закрепил (тот, кто нажал «Применить» или сделал волевую правку). |
| `officialContentReason` | enum: `'vote' \| 'admin' \| 'initial'` | Почему текст стал официальным: победил в голосовании, закреплён волей админа, начальная заливка. |
| `officialContentVariantId` | ObjectId? | Если `reason='vote'` — ссылка на победивший вариант. |

### 4.4 DocumentBlockVariant
Хранится в коллекции `documentBlockVariants`.

| Поле | Тип | Описание |
|---|---|---|
| `_id` | ObjectId | |
| `documentId` | ObjectId | |
| `blockId` | string | uuid блока (см. §4.3). |
| `content` | string (rich text) | Предложенный текст. |
| `references` | Reference[] | Ссылки-обоснования (см. §4.5). |
| `proposedBy` | UserId | Автор варианта. |
| `proposedAt` | Date | |
| `votingDeadline` | Date | `proposedAt + document.votingDurationHours`. На вариант голосуют до этого момента. |
| `status` | enum: `'open' \| 'closed-winner' \| 'closed-not-winner' \| 'applied' \| 'withdrawn'` | Жизненный цикл варианта (§13). |
| `rating` | number | Текущий рейтинг варианта (сумма голосов). Денормализован для скорости. |
| `appliedAt` | Date? | Когда применён. |
| `appliedBy` | UserId? | Кто применил (для ручного и волевого режима). |
| `costPaid` | number | Сколько заслуг было заплачено при предложении варианта (§14). |
| `deleted`, `deletedAt` | boolean / Date | Soft delete. |

**Индексы:**
- `(documentId, blockId, status)` — выдача активных вариантов на блок.
- `(documentId, blockId, rating desc)` — поиск победителя.
- `votingDeadline` — для cron-фонового закрытия голосований.

### 4.5 Reference (вложенная в DocumentBlockVariant)

| Поле | Тип | Описание |
|---|---|---|
| `id` | string (uuid) | |
| `url` | string | URL (валидируется как URL). |
| `summary` | string | Короткое описание (1–2 строки, лимит ~280 символов). |
| `stance` | enum: `'pro' \| 'con'` (опционально) | За или против формулировки. |

### 4.6 Голоса за варианты
Используется **существующая** механика голосования (см. `business-content.mdc`, секция «Comment Data Model (Vote as comment with weight)»). Не создаём новую коллекцию.

**Привязка к существующей системе:**
- `Vote.targetType` — добавить новое значение `'document-variant'`.
- `Vote.targetId` — `DocumentBlockVariant._id`.
- `Vote.direction` — `'up' \| 'down'`.
- `Vote.amountQuota` + `Vote.amountWallet` — суммарный вес = вклад в `variant.rating`.
- `Vote.comment` — **обязателен** и для `up`, и для `down` (см. §15.2).
- `Vote.communityId` — сообщество/проект, к которому принадлежит документ.

**Денежные правила** (по аналогии с обычными комментариями к постам, см. `business-merits.mdc` → "Voting for Content"):
- Upvote: quota → wallet (приоритет квоты).
- Downvote: wallet only.
- Self-vote (за свой вариант): запрещён по умолчанию.
- Источник средств (quota/wallet) подчиняется `votingSettings.currencySource` сообщества.
- Routing кошелька: для приоритетных сообществ — global wallet; иначе — wallet сообщества. Это **уже работает** для голосов и должно автоматически распространиться на новый `targetType`.

## 5. Настройки сообщества/проекта (новые поля)

Добавляются в **существующий** блок `Community.settings` (см. `business-communities.mdc` → "Community Settings").

| Поле | Тип | Default | Описание |
|---|---|---|---|
| `documentsMode` | enum: `'off' \| 'visionOrDescriptionOnly' \| 'all'` | `'visionOrDescriptionOnly'` | Режим работы совместных документов. См. §5.2. |
| `documentCreators` | enum: `'admins' \| 'members'` | `'admins'` | Кто может создавать кастомные документы (только при `documentsMode='all'`). |
| `documentVariantCost` | number? | `null` (= наследует `settings.postCost`) | Стоимость предложить вариант блока. Если задано — переопределяет дефолт. |
| `documentVotingDurationHours` | number | `48` | Дефолтная длительность голосования для **новых** документов в этом сообществе. Уже созданные документы хранят свою длительность в `Document.votingDurationHours`. |
| `documentDefaultMode` | enum: `'manual' \| 'auto'` | `'manual'` | Дефолтный режим применения победителя для новых документов. |
| `documentAutoApplyTimerHours` | number | `48` | Для `documentDefaultMode='auto'` — через сколько часов после старта голосования автоприменять победителя. Используется только если на самом документе `mode='auto'`. |

### 5.1 Миграция существующих сообществ/проектов
- **Все существующие** сообщества и проекты получают `documentsMode='visionOrDescriptionOnly'` миграцией.
- Для каждого существующего сообщества с непустым полем «Образ Будущего» создаётся `Document` с `type='imageOfFuture'`:
  - одна секция, без заголовка;
  - один блок типа `paragraph`;
  - `officialContent` = существующий текст ОБ;
  - `officialContentReason = 'initial'`;
  - `officialContentSetBy` = текущий админ сообщества (lead) или системный bootstrap-юзер, если lead не определён.
- Аналогично для каждого существующего проекта с непустым «Описание проекта» — создаётся `Document` с `type='description'`.
- **Старое поле текста ОБ/Описания не удаляется** — оно становится «зеркалом» официального текста документа (см. §5.3). Это нужно для обратной совместимости с существующими местами в UI, где этот текст уже читается.

### 5.2 Семантика режимов
| Режим | UI: вкладка «Документы» | UI: редактирование ОБ/Описания | UI: создание кастомных доков |
|---|---|---|---|
| `'off'` | Скрыта | Старое поведение (простой текст, admin-only) — текст из `Document` всё равно подтягивается, но интерфейс предложения вариантов скрыт. Админ может редактировать прямо через «волевое решение» (см. §12.3). | Запрещено |
| `'visionOrDescriptionOnly'` (default) | **Скрыта** | Совместный режим: интерфейс «Предложить вариант», голосование, применение. | Запрещено |
| `'all'` | **Видна** в навигации | Совместный режим (как в `visionOrDescriptionOnly`). | Разрешено в соответствии с `documentCreators` |

### 5.3 Синхронизация официального текста с существующим полем ОБ/Описания
Когда у `ОБ-документа` / `Описание-документа` меняется `officialContent` любого блока (по победе варианта или волевым решением), бекенд **должен** обновлять существующее поле текста ОБ/Описания в сообществе/проекте.

- **Формат записи в старое поле**: конкатенация всех `officialContent` всех блоков всех секций в порядке `section.order`, `block.order`, с разделителями `\n\n` между блоками и `\n\n# {section.title}\n\n` перед секцией с заголовком (если `section.title` непустой). Это даёт читаемый plain-text/markdown, совместимый с тем, что было раньше.
- Если в будущем планируется хранить структурированный документ напрямую — это отдельная задача, в скоуп не входит.

## 6. UI: вкладка «Документы» в ленте

### 6.1 Где появляется
- Сообщество: текущий набор `Посты / Проекты / Ивенты / Мы на бирже` → становится `Посты / Проекты / Ивенты / Документы / Мы на бирже` (5 вкладок).
- Проект: текущий набор `Посты / Ивенты / Мы на бирже` → становится `Посты / Ивенты / Документы / Мы на бирже` (4 вкладки).
- Видимость вкладки определяется `documentsMode === 'all'` (см. §5.2).

### 6.2 Мобильная навигация
Текущая горизонтальная навигация на мобильных уже использует горизонтальный скролл (Курсор проверяет реализацию в `components/`/`features/`).
- **Решение:** оставить горизонтальный скролл, добавить вкладку без переноса. Никаких «бургеров»/«more»-меню.
- Если в результате тестирования окажется, что 5 вкладок не помещаются и часть скрывается под краем — Курсор смотрит существующий паттерн в коде. Если он не предполагает скролл — нужно либо ввести скролл, либо сократить лейблы на узких экранах (≤ 380px) до иконок. **Решение принимает Курсор/Дима**, с предпочтением скролла как наименее ломающего вариант.

### 6.3 Что показывается на вкладке
Список документов сообщества/проекта (только активные, без soft-deleted):

| Тип | Показывается всегда? | Закреплён? |
|---|---|---|
| `imageOfFuture` (для сообществ) | Да | Да, всегда первым в списке. |
| `description` (для проектов) | Да | Да, всегда первым в списке. |
| `custom` | Да, в порядке `createdAt desc` | Нет |

Карточка документа в списке:
- Иконка типа (📜 для ОБ, 📋 для Описания, 📄 для кастома).
- Заголовок (`title`).
- Краткий превью первого блока первой секции (первые ~120 символов `officialContent`).
- Метаданные: дата последнего обновления, число активных вариантов на блоках (плашка «Есть N открытых предложений»).
- Клик → открывает страницу документа (§7).

### 6.4 Кнопка «Создать документ»
- Видна только при `documentsMode='all'` И когда текущий пользователь имеет право (по `documentCreators`):
  - `documentCreators='admins'` — только админ/lead сообщества.
  - `documentCreators='members'` — любой участник сообщества/проекта (не гость, не неподтверждённый).
- Открывает диалог создания (см. §11).

### 6.5 «Только для ОБ/Описания» (`visionOrDescriptionOnly`)
Если вкладка скрыта, доступ к ОБ-документу всё равно нужен:
- **Сообщество:** на главной странице сообщества (там, где сейчас показывается ОБ как текст) рядом с текстом появляется кнопка/ссылка «Открыть совместный режим» → ведёт на страницу документа (§7).
- **Проект:** аналогично, рядом с «Описание проекта» — кнопка «Открыть совместный режим».
- Альтернативный взгляд: текст ОБ/Описания сразу становится «кликабельным заголовком/превью документа», который при тапе открывает полноценный документ. Курсор выбирает менее интрузивный вариант в соответствии с существующим UI.

## 7. UI: страница документа

### 7.1 Структура страницы
- Заголовок документа (`title`, не редактируемый для ОБ/Описания; редактируемый для кастомов админом).
- Метаданные: тип, последнее обновление, режим (manual/auto), длительность голосования.
- Кнопка «Настройки документа» (видна автору и админу) — открывает диалог редактирования метаданных (`mode`, `votingDurationHours`, `variantCost`, `allowDownvotes`).
- Кнопка «История» (см. §19).
- Тело документа: секции → блоки.

### 7.2 Отображение блока
Для каждого блока показывается:
1. **Официальный текст** (`officialContent`), отрендеренный по `blockType`.
2. **Метка-плашка с источником**: 
   - «Принят голосованием N сообществом-X» (если `reason='vote'`)
   - «Закреплён админом @username» (если `reason='admin'`)
   - «Начальный текст» (если `reason='initial'`)
3. Кнопка **«Предложить вариант»** (доступна всем участникам сообщества, при условии оплаты — см. §14).
4. Кнопка **«Варианты (N)»** — раскрывает панель с открытыми вариантами на этот блок.
5. Для админа/автора документа: меню «…» с действиями:
   - Применить произвольный текст (волевое решение, §12.3).
   - Закрыть голосование вручную (если есть открытые варианты).
   - Удалить вариант (если есть нарушающие правила).

### 7.3 Панель «Варианты блока»
Раскрывается под блоком. Сортировка по `rating desc`.

Каждый вариант:
- Текст (`content`), отрендеренный по `blockType`.
- Автор варианта, дата предложения.
- Дедлайн голосования (`votingDeadline`), таймер «осталось N часов».
- Рейтинг варианта (`rating`), число голосов «за» и «против».
- Кнопки «Голосовать за / против» (см. §15).
- Список ссылок-обоснований (с pro/con маркерами).
- Кнопка «Подробнее» — открывает обсуждение варианта (комментарии = существующая система комментариев, привязанных к `targetType='document-variant'`, если применимо; иначе используем существующий механизм голосования с обязательным комментарием).
- Плашка статуса: «Открыто», «Победитель — ожидает применения», «Применён», «Голосование закрыто, не победил», «Отозван».
- Для автора варианта (пока статус `'open'`): кнопка «Отозвать вариант» (`withdrawn`). Заплаченная стоимость **не возвращается** (см. §14.5).
- Для автора/админа документа: кнопка «Применить вариант» (видна для победителя в `manual` режиме).

### 7.4 Структурное редактирование (секции/блоки)
Для **MVP** (ОБ/Описание) минимум 1 секция и 1 блок — этого достаточно. Структурное редактирование возможно, но скрыто за правом «автор документа / админ»:
- Добавить/удалить/переименовать секцию.
- Добавить/удалить/переместить блок.
- Изменить `blockType` блока.

**Что происходит с активными вариантами при удалении блока:** все открытые варианты на этот блок становятся `status='withdrawn'`, голоса остаются в Vote-коллекции для аудита. Уведомлений нет.

**Что происходит при удалении блока с применённым текстом:** разрешено только для админа, требует подтверждения «Этот блок содержит официальный текст, потеряете его — продолжить?».

## 8. Жизненный цикл ОБ-документа / Описание-документа

### 8.1 Создание
- При создании нового сообщества: создаётся `Document` с `type='imageOfFuture'`, одна секция, один пустой блок типа `paragraph`. `documentsMode` сообщества по умолчанию = `'visionOrDescriptionOnly'`.
- При создании нового проекта (`typeTag='team'`): создаётся `Document` с `type='description'`, одна секция, один пустой блок.
- Создатель сообщества/проекта становится `createdBy` документа и может его сразу заполнить (см. §12.3 — волевое решение, без оплаты для начальной заливки `reason='initial'`).

### 8.2 Жизненный цикл
- Текст редактируется только через:
  - **волевое решение админа/автора** (§12.3) — мгновенно обновляет `officialContent`;
  - **применение варианта** (§12.1, §12.2) — после голосования.
- Документ не имеет «закрытого» состояния — он живёт всё время, пока живёт сообщество/проект.
- При архивировании/удалении сообщества/проекта — документ помечается `status='archived'` или soft-deleted.

### 8.3 Запрет на удаление
- ОБ-документ нельзя удалить, пока существует сообщество.
- Описание-документ нельзя удалить, пока существует проект.
- Кастомные документы — можно (см. §11.4).

## 9. Жизненный цикл кастомного документа (`type='custom'`)

### 9.1 Создание
Доступно при `documentsMode='all'` и согласно `documentCreators`.
- Открывается диалог: title, начальная структура (минимум 1 секция, 1 блок), настройки (`mode`, `votingDurationHours`, `variantCost`, `allowDownvotes`).
- При создании автор может сразу заполнить блоки своим текстом — это `reason='initial'`, бесплатно.

### 9.2 Жизненный цикл
- Аналогичен ОБ-документу, с тем отличием, что админ может архивировать или удалить.

### 9.3 Архивирование / удаление
- Автор документа или админ сообщества может архивировать (`status='archived'`) — документ исчезает из основного списка, но доступен в архиве.
- Soft delete (`deleted=true`) — доступен админу. Восстановление — по аналогии с публикациями (`publications.restore`).
- Hard delete — только админу, подтверждение.

### 9.4 Что происходит при смене `documentsMode` с `'all'` на `'visionOrDescriptionOnly'` или `'off'`
- Существующие кастомные документы **не удаляются**, но скрываются из UI (вкладка «Документы» исчезает).
- Активные варианты на их блоках продолжают накапливать голоса до своего `votingDeadline` (фоновый cron не зависит от настроек видимости).
- При возвращении в `'all'` — всё восстанавливается.

## 10. Флоу: предложить вариант блока

### 10.1 Условия
- Пользователь — участник сообщества/проекта.
- На этом блоке от этого пользователя **нет активного открытого варианта** (один пользователь — один открытый вариант на блок).
- У пользователя достаточно заслуг для оплаты `document.variantCost` (см. §14).
- `documentsMode != 'off'`.

### 10.2 Шаги
1. Пользователь нажимает «Предложить вариант» под блоком.
2. Открывается диалог:
   - Поле текста с WYSIWYG (рендерится в соответствии с `blockType`).
   - Поле «Ссылки-обоснования» (необязательно): можно добавить N ссылок (рекомендуемый лимит — 10), у каждой URL, summary (≤ 280 символов), опциональный `stance: 'pro' \| 'con'`.
   - Информационный блок: «Предложение варианта стоит **{variantCost}** заслуг. Эта сумма сгорает как комиссия и не возвращается». Если у пользователя недостаточно заслуг — кнопка «Отправить» неактивна, показывается баланс.
3. Пользователь нажимает «Отправить».
4. Бекенд:
   - Списывает `variantCost` заслуг из global wallet пользователя (источник средств — по аналогии с `publications.create`, см. `business-merits.mdc`).
   - **Заслуги сгорают** (комиссия, не зачисляются никому).
   - Создаёт `DocumentBlockVariant` со `status='open'`, `proposedAt=now`, `votingDeadline=now + document.votingDurationHours`.
   - Возвращает успех.
5. UI обновляет панель «Варианты блока», новый вариант появляется первым.

### 10.3 Валидация
- `content` не пустой, длина не превышает разумного лимита (предложение: 5000 символов).
- Каждая ссылка — валидный URL.
- Если у блока уже есть открытый вариант от того же пользователя — отказ (`'You already have an open variant on this block'`).

## 11. Создание кастомного документа

### 11.1 Условия
- `documentsMode='all'`.
- Пользователь имеет право по `documentCreators` (admins/members).

### 11.2 Шаги
1. Пользователь на вкладке «Документы» нажимает «Создать документ».
2. Открывается диалог:
   - Поле «Название» (required, ≤ 200 символов).
   - Заготовка структуры: 1 секция «Общее» с 1 блоком типа `paragraph`. Можно сразу добавить секции/блоки.
   - Поля настроек документа (с дефолтами из сообщества):
     - `mode` (manual/auto).
     - `votingDurationHours` (число).
     - `variantCost` (число).
     - `allowDownvotes` (boolean).
   - Кнопка «Создать».
3. Бекенд создаёт `Document`, автором становится текущий пользователь (`createdBy`).
4. Открывается страница нового документа (§7), автор может сразу заполнить блоки (`reason='initial'`, бесплатно).

### 11.3 Создание кастомного дока бесплатно
Создание самого документа не стоит заслуг. Заслуги начинают тратиться только когда кто-то предлагает варианты блоков (§14).

### 11.4 Удаление / архивирование кастомного дока
- Автор документа: может архивировать.
- Админ сообщества: может архивировать, soft-delete, permanent delete.
- Удаление документа автоматически переводит все открытые варианты в `status='withdrawn'`.

## 12. Применение победителя

Реализуем **ручной режим по умолчанию (1A)**. **Авто-режим (1B)** реализован тоже — настройка `Document.mode`. Решение бизнеса: §6 исходного ТЗ, ответ 1.

### 12.1 Ручное применение (`mode='manual'`, дефолт)
**Когда становится возможным:**
- Голосование по блоку закрылось (см. §13).
- Среди голосов есть положительный лидер (`rating > 0`).
- Лидер получил статус `status='closed-winner'`.

**Действия:**
- В UI на варианте-победителе появляется плашка «Победитель — ожидает применения» (видна всем).
- Автор документа или админ сообщества/проекта видит кнопку «Применить вариант».
- При нажатии:
  - `Block.officialContent` ← `variant.content`.
  - `Block.officialContentSetAt = now`.
  - `Block.officialContentSetBy = currentUser`.
  - `Block.officialContentReason = 'vote'`.
  - `Block.officialContentVariantId = variant._id`.
  - `Variant.status = 'applied'`, `appliedAt = now`, `appliedBy = currentUser`.
  - Все прочие открытые варианты на этом блоке → `status='closed-not-winner'`.
  - Если документ — ОБ или Описание, обновляется зеркало в существующем поле текста (§5.3).
- Если автор/админ не применил победителя — он остаётся в `closed-winner` неограниченно долго, пока кто-то новый не предложит вариант (новые варианты открывают новый цикл — см. §13.5).

### 12.2 Авто-применение (`mode='auto'`)
- Правило автопринятия: **по времени** (см. §6 исходного ТЗ, ответ 3). 
- Конкретно: через `Document.votingDurationHours` (= конец цикла голосования) **автоматически** применяется лидер, если `rating > 0`.
- Реализация: cron-задача (по аналогии с post-closing cron, см. `architecture.mdc` → `domain/services/`).
- При автоприменении `Block.officialContentSetBy = systemUserId` (специальный системный юзер), `Block.officialContentReason = 'vote'`. В истории и в плашке указывается «Принято автоматически по голосованию».

### 12.3 Волевое решение (любой режим)
Автор документа или админ сообщества может в любой момент:
- **Применить любой открытый вариант** (минуя голосование) — кнопка на варианте «Применить волей админа». Действует как §12.1, но `reason='admin'`, остальные варианты → `closed-not-winner`.
- **Применить произвольный текст**, не выбирая ни один вариант — кнопка на блоке «Изменить текст волевым решением». Открывает редактор, после сохранения:
  - `officialContent` ← новый текст,
  - `reason='admin'`, `setBy=currentUser`,
  - все открытые варианты на блоке → `closed-not-winner`.

Волевое решение не требует оплаты — это admin-action.

### 12.4 Если рейтинг победителя ≤ 0
- В ручном режиме: кнопка «Применить» **не показывается**. Админ может либо применить волей (§12.3), либо игнорировать.
- В авто-режиме: автоприменение **не срабатывает**. Цикл закрывается, статус всех — `closed-not-winner`. Логируется причина «no positive winner».

## 13. Закрытие голосования по блоку

Реализуем **таймер** (см. §7 исходного ТЗ, ответ 1). Уровень настройки — **документ** (см. §7, ответ 2).

### 13.1 Когда стартует таймер
- В момент создания первого открытого варианта на блок (`status='open'`).
- `votingDeadline` варианта = `proposedAt + document.votingDurationHours`.

### 13.2 Что происходит при наступлении дедлайна
Cron (раз в N минут, рекомендуемая частота — 5 минут) проверяет все варианты со `status='open'` и `votingDeadline <= now`:
- Берёт все варианты на этом блоке со `status='open'` с `votingDeadline <= now`.
- Определяет лидера по `rating` (если несколько с одинаковым рейтингом — берёт самый ранний по `proposedAt`).
- Лидер → `status='closed-winner'` (если `rating > 0`) или `status='closed-not-winner'` (если `rating <= 0`).
- Остальные → `status='closed-not-winner'`.
- Если `Document.mode='auto'` и есть `closed-winner` — сразу применяется (§12.2).

### 13.3 Несколько вариантов с разными дедлайнами
Каждый вариант имеет свой `votingDeadline`. Логика «закрытия цикла на блок» работает так:
- Цикл голосования по блоку считается **открытым**, пока есть хотя бы один вариант со `status='open'`.
- Когда последний открытый вариант доходит до своего дедлайна — закрывается общий цикл по блоку: определяется лидер среди всех `open`-вариантов, которые сейчас закрываются (их деадлайны могут совпасть или быть в разное время; для упрощения — закрытие происходит индивидуально по дедлайну каждого варианта; победитель определяется по сравнению со всеми вариантами на блок в момент закрытия каждого).

**Упрощённая модель (рекомендуется):** все варианты, появившиеся в течение `votingDurationHours` от **первого открытого** варианта на блок, считаются одной волной. Дедлайн волны = `firstVariantInWave.proposedAt + votingDurationHours`. В момент дедлайна — все они закрываются разом, определяется победитель волны. Это упрощает UI и UX. **Принимаем эту модель.**

→ Уточнение схемы: вместо `Variant.votingDeadline` использовать `Block.currentWaveStartedAt` (Date?) на блоке. Дедлайн волны = `currentWaveStartedAt + Document.votingDurationHours`. Когда волна закрывается — `currentWaveStartedAt` сбрасывается в null.

### 13.4 Ручное закрытие админом
Админ может в любой момент нажать «Закрыть голосование сейчас» на блоке. Это форсирует закрытие текущей волны (логика §13.2 применяется немедленно).

### 13.5 Открытие новой волны
После закрытия волны на блоке (любым способом) — следующий открытый вариант стартует **новую** волну. Между волнами блок может находиться в состояниях:
- `closed-winner есть, ожидает применения` (manual mode), 
- `applied` (если применили),
- `closed-not-winner у всех` (никто не победил, либо победитель не был применён, либо рейтинг был ≤ 0).

Новая волна не сбрасывает предыдущие закрытые варианты — они остаются в БД с финальным статусом, видны в истории (§19).

## 14. Антиспам: стоимость предложить вариант

Решение бизнеса (§9 исходного ТЗ): **платный барьер, сумму выставляет админ, по умолчанию равна `settings.postCost`, сгорает как комиссия**.

### 14.1 Откуда берётся стоимость
- На уровне документа: `Document.variantCost`.
- При создании документа: `variantCost` инициализируется из `Community.settings.documentVariantCost` (если задан), иначе из `Community.settings.postCost`.
- Админ сообщества может изменить `Community.settings.documentVariantCost` — это поменяет дефолт для **новых** документов, но не для уже созданных.
- Автор документа может изменить `Document.variantCost` через настройки документа (см. §7.1).

### 14.2 Кому уходят деньги
**Никому. Сгорают.** Эквивалент платы за post creation (см. `business-merits.mdc` → "Fee payment is always from global merits"). Списываются с `GLOBAL_COMMUNITY_ID` wallet пользователя.

### 14.3 Источник средств у пользователя
- По аналогии с `publications.create`: wallet fee debited from `GLOBAL_COMMUNITY_ID`.
- Опционально, если `settings.canPayPostFromQuota=true` — квота может покрыть часть.
- Если средств не хватает — вариант не создаётся, ошибка `INSUFFICIENT_FUNDS`.

### 14.4 Логирование
- Создаётся запись в существующей системе транзакций (по аналогии с `publications.create` charge).
- `costPaid` хранится на самом варианте для аудита.

### 14.5 Отзыв варианта
- Если автор отзывает вариант (`status='withdrawn'`) — **деньги не возвращаются**.
- Если админ удаляет вариант — **деньги не возвращаются**.
- Если волна закрылась и вариант проиграл — **деньги не возвращаются**.

> Бизнес-логика: оплата — это барьер от спама, а не «ставка». Никаких возвратов.

## 15. Голосование за варианты

Решение бизнеса (§8 исходного ТЗ): **«за» и «против» оба нужны; цена обычная; комментарий обязателен для обоих**.

### 15.1 Где живут голоса
Используем существующую сущность `Vote` (см. §4.6). Никаких новых коллекций.

### 15.2 Обязательный комментарий
- При голосе `up` или `down` **обязательно** заполнить поле `comment` (минимум 1 символ; рекомендуемый минимум — 10 символов и максимум — 500).
- Валидация на бекенде. На фронте — UX-валидация поля «Комментарий», кнопка «Проголосовать» неактивна, пока поле пустое.
- Это отличие от обычных votes на публикации, где комментарий часто опционален. В тRPC процедуре для `targetType='document-variant'` валидация форсирует наличие комментария.

### 15.3 Цена голоса
- Стоимость голоса = `amountQuota + amountWallet` (как в обычных votes). Минимум 1 заслуга.
- Источник: 
  - Upvote: quota → wallet (приоритет квоты).
  - Downvote: wallet only.
  - Self-vote (за свой вариант): **запрещён**.
- Источник также подчиняется `votingSettings.currencySource` сообщества.

### 15.4 Расчёт рейтинга варианта
`Variant.rating = sum(up votes amount) - sum(down votes amount)`.

Денормализован в поле `Variant.rating` для скорости. Обновляется атомарно при каждом голосе.

### 15.5 Изменение/отзыв голоса
- Стандартная механика: пользователь может **withdraw** свой голос (как `votes.withdrawFromVote`). Логика возврата заслуг — по существующим правилам (см. `business-merits.mdc`).
- После закрытия волны (§13) голосовать или отзывать голоса нельзя.

### 15.6 Запрет минусования
Если `Document.allowDownvotes=false` — кнопка «Голосовать против» скрыта; попытка downvote через API → ошибка `DOWNVOTES_DISABLED`.

> Бизнес-решение: по умолчанию `allowDownvotes=true` (минусы разрешены). Это можно отключить на уровне отдельного документа.

## 19. История документа

### 19.1 Что хранится
**Не вводим отдельную коллекцию `DocumentHistory`** в MVP. Используем существующие сущности:
- Применённые варианты остаются в `DocumentBlockVariant` с финальными статусами (`applied`, `closed-not-winner`, `closed-winner`).
- Для волевых правок и initial-заливок — заводим легковесный лог `DocumentBlockHistoryEntry` или используем поле `editHistory` (по аналогии с публикациями, см. `business-content.mdc` → "Editing" → `editHistory`).

**Предлагаемая схема `editHistory`** (вложенный массив на `Block`):
```
[
  { changedAt: Date, changedBy: UserId, reason: 'initial'|'vote'|'admin', 
    variantId?: ObjectId, previousContent: string }
]
```
- При каждом изменении `officialContent` добавляется запись.
- `previousContent` хранит то, что было ДО изменения. Это даёт минимальный аудит без UI отката.

### 19.2 UI «История»
- Кнопка «История» на странице документа (§7.1) или на каждом блоке.
- Открывает таймлайн изменений по документу/блоку:
  - Дата, автор, причина (`vote`/`admin`/`initial`), ссылка на вариант (если был).
  - Превью предыдущего и нового текста.
- Откат **не реализуем** в MVP (можно в v2.0).

### 19.3 Запретенные операции
- Удалить запись истории — нельзя никому, кроме superadmin (через прямой доступ к БД, не через UI).

## 20. API (tRPC endpoints — предложение)

Размещение: `api/apps/meriter/src/trpc/routers/document.router.ts` (новый файл) + регистрация в `trpc/router.ts`. Сервис: `api/apps/meriter/src/domain/services/document.service.ts` (новый). Schema: `api/apps/meriter/src/domain/models/document/document.schema.ts`.

### 20.1 Документы
- `documents.getByCommunity({ communityId })` — все активные документы сообщества/проекта.
- `documents.getById({ id })` — документ со всеми секциями и блоками.
- `documents.getOfficialDocumentByType({ communityId, type })` — получить ОБ-документ или Описание-документ конкретного сообщества/проекта.
- `documents.create({ communityId, title, sections, ... })` — создание кастомного дока.
- `documents.updateMeta({ id, title?, mode?, votingDurationHours?, variantCost?, allowDownvotes? })`.
- `documents.archive({ id })` / `documents.unarchive({ id })`.
- `documents.delete({ id })` (soft).
- `documents.restore({ id })`.
- `documents.permanentDelete({ id })`.

### 20.2 Структура (секции/блоки)
- `documents.addSection({ documentId, title?, order? })`.
- `documents.updateSection({ documentId, sectionId, title?, order? })`.
- `documents.removeSection({ documentId, sectionId })`.
- `documents.addBlock({ documentId, sectionId, blockType, content?, order? })`.
- `documents.updateBlock({ documentId, sectionId, blockId, blockType?, order? })` — без изменения `officialContent` (для этого `applyAdminOverride`).
- `documents.removeBlock({ documentId, sectionId, blockId })`.
- `documents.applyAdminOverride({ documentId, blockId, newContent })` — волевое решение (§12.3, произвольный текст).

### 20.3 Варианты
- `documentVariants.getByBlock({ documentId, blockId, status?: 'open'|'all' })`.
- `documentVariants.propose({ documentId, blockId, content, references? })` — создаёт вариант, списывает `variantCost`.
- `documentVariants.withdraw({ variantId })` — автор отзывает.
- `documentVariants.delete({ variantId })` — админ удаляет.
- `documentVariants.apply({ variantId, byVote: boolean })` — применить вариант (manual-mode победителя или волей). `byVote=true` — только если статус `closed-winner`. `byVote=false` — волевое.
- `documentVariants.closeVotingWaveOnBlock({ documentId, blockId })` — админ форсирует закрытие волны (§13.4).

### 20.4 Голосование (расширение существующего)
**Не плодим новый эндпоинт.** Расширяем существующий `votes.cast` (или эквивалент) принимать `targetType='document-variant'`.

Логика валидации в сервисе голосов:
- Если `targetType='document-variant'`, **обязательный** `comment` (§15.2).
- Проверка `Document.allowDownvotes` для downvote.
- Проверка, что вариант имеет `status='open'`.
- Запрет self-vote (автор варианта).

### 20.5 Настройки сообщества
Расширение существующего `communities.updateSettings` (или эквивалента) — принимает новые поля (§5).

### 20.6 Cron / фоновое
- `domain/services/document-voting-cron.service.ts` (по аналогии с post-closing cron):
  - раз в 5 минут: ищет блоки с истёкшим дедлайном волны → закрывает (§13.2);
  - если `Document.mode='auto'` — применяет победителя (§12.2).

### 20.7 Нотификации (минимум)
Используем существующую систему нотификаций (см. `business-content.mdc` / `business-merits.mdc`). Новые типы:
- `document_variant_won` — автору варианта, когда его вариант стал `closed-winner`.
- `document_variant_applied` — автору варианта, когда вариант стал официальным.
- `document_block_admin_override` — авторам всех открытых вариантов на блоке, когда админ закрыл их волевым решением (опционально, может быть отложено).

| Нумерованный список | ✅ |

### 22.3 Решения бизнеса (§11 исходного ТЗ)
1. **Изображения** — НЕТ (упрощаем UI и storage).
2. **Код-блок** — НЕТ (для пользовательского текста не нужен).
3. **Google Docs paste** — НЕТ, достаточно базовой вставки plain text/markdown.

### 22.4 Реализация
- Рекомендуемая библиотека: **TipTap** (на ProseMirror) — современная, расширяемая, поддерживает все нужные пункты «из коробки», даёт богатый формат хранения (JSON или HTML).
- Альтернативы: Slate.js, Lexical (Meta), Quill. Курсор выбирает по совместимости с React 19 + Next.js 16 + текущим стеком.
- **Формат хранения:** Markdown (предпочтительно, простой, читабельный в БД, легко рендерить везде) **или** ProseMirror JSON / HTML (если выбран TipTap). Курсор фиксирует выбор в PRD.
- **Sanitization:** обязательная серверная очистка от XSS (например, `DOMPurify` на сервере при сохранении HTML; или markdown как text-only).
- Компонент: `web/src/components/molecules/RichTextEditor.tsx` (или эквивалент).
- Прежние «простые текстовые поля» в постах **не мигрируем** автоматически — они остаются текстом, но новые посты/правки могут уже использовать WYSIWYG. Решение «глобальная замена textarea → WYSIWYG» — за пределами этой фичи, можно сделать инкрементально.

### 22.5 Скоуп WYSIWYG в этом релизе
- **Обязательно**: WYSIWYG используется в полях вариантов блоков (§10.2) и в полях официального текста блока (§7.2 рендер + админ-редактор §12.3).
- **Желательно**: WYSIWYG доступен также для создания новых постов и для описания сообществ/проектов.
- **Не входит**: ретроактивная миграция старых текстов.

## 23. Производительность и масштаб

- Документ обычно содержит ≤ 10 секций × ≤ 20 блоков = ≤ 200 блоков. Документы — мелкие документы.
- Вариантов на блок одновременно — единицы, редко десятки. Лимит «1 активный вариант от пользователя на блок» (§10.1) — серьёзный регулятор.
- Запросы:
  - Чтение документа: один документ + список открытых вариантов (по `documentId, status='open'`).
  - Голос: один insert в Vote + один update `Variant.rating`.
- Узких мест на старте не ожидается. Стандартные индексы (см. §4.4).

## 25. Открытые вопросы / риски

> Эти пункты не блокируют разработку, но Курсор должен зафиксировать их в `agent-brief.md` для последующего обсуждения.

1. **Точное имя поля «Образ Будущего» в коде**: уточняется при анализе схемы community. Возможно, это `description`, возможно — отдельное поле. От этого зависит §5.3 (синхронизация).
2. **Что делать с `future-vision`-сообществом (специальное приоритетное сообщество)?** В нём, возможно, не нужно создавать ОБ-документ (это само и есть «образ будущего» как сущность). Курсор проверяет и фиксирует решение.
3. **Mobile-навигация с 5 вкладками** — может потребоваться экспериментальная проверка. Если горизонтальный скролл уже есть, добавление вкладки тривиально; если нет — нужен fallback.
4. **Уведомления (§20.7)** — можно отложить часть типов на v2.0 (особенно `document_block_admin_override`).
5. **Откат изменений** (UI отката из истории) — НЕ входит в MVP, можно реализовать позднее, на чистой данной модели §19.
6. **Импорт/экспорт документов** — НЕ входит.
7. **Поведение при изменении структуры (удаление блока) во время активной волны** — выбрано: `withdrawn` для всех вариантов, без возврата заслуг (§7.4). Если бизнес позднее захочет «защитить» голосование от структурных изменений — добавить флаг «заморозка блока на время волны».
8. **Глобальные документы кросс-сообщества** — НЕТ в скоупе. Каждый документ принадлежит ровно одному сообществу/проекту.
9. **Полнотекстовый поиск по документам** — НЕ в скоупе MVP.
10. **Тестовые данные для демо** — Курсор/Дима готовят набор демо-документов после реализации (см. упоминание demo-content в `projects-conversation-analysis.md` §1.1).
