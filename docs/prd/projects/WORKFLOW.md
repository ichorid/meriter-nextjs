# Workflow: Реализация фичи «Проекты»

## Структура работы

```
Sprint 1 → Review → Fix → ✅ → Sprint 2 → Review → Fix → ✅ → ... → Sprint 6 → ✅
```

Каждый спринт = **отдельный агент Cursor** (новый чат). Контекст между спринтами передаётся через completion reports.

При работе над фичей «Проекты» веди **progress log** (см. правило в `.cursor/rules/progress-log.mdc`): создай/обновляй `docs/prd/projects-progress.md` (или файл по правилу) и при смене спринта прикладывай его к новому чату для непрерывности контекста.

```
├── SPRINT-6-PRD.md

    ├── SPRINT-2-REPORT.md  # ...
    └── SPRINT-6-REPORT.md
```

## Для каждого спринта

### 1. Открой новый агент Cursor

### 2. Скопируй промпт из `prompts/sprint-N.md`

Промпт уже содержит:
- Ссылки на файлы для загрузки
- Инструкции по автономной работе
- Формат отчёта

### 3. Cursor работает автономно

Ты видишь работу. Не вмешивайся, пока не увидишь отчёт.

### 4. Cursor выдаёт отчёт

Формат:
- ✅ Сделано (конкретно что)
- ⚠️ Решения, принятые по ходу
- ❌ Не удалось / заблокировано
- 📋 Чеклист для проверки (что ты должен руками проверить)

### 5. Ты проверяешь

Смотришь чеклист, проверяешь ключевые точки. **Перед переходом к следующему спринту** убедись, что отчёт содержит все обязательные блоки (см. единый шаблон отчёта ниже; для Sprint 3 — обязательно «Где перехвачен withdraw» и «Где merit resolver override»). Если блоков нет — попроси Cursor дополнить отчёт. Если ок → переходи к следующему спринту. Если нет → скажи что не так, Cursor исправит.

### 6. Cursor обновляет report

После фиксов — финальный report. Ты подтверждаешь. Переход.

## Что загружать в Cursor для каждого спринта

| Sprint | Файлы |
|--------|-------|
| 1 | `SPRINT-1-PRD.md` |
| 2 | `SPRINT-2-PRD.md` + `reports/SPRINT-1-REPORT.md` |
| 3 | `SPRINT-3-PRD.md` + `reports/SPRINT-2-REPORT.md` |
| 4 | `SPRINT-4-PRD.md` + `reports/SPRINT-3-REPORT.md` |
| 5 | `SPRINT-5-PRD.md` + `reports/SPRINT-4-REPORT.md` |
| 6 | `SPRINT-6-PRD.md` + `reports/SPRINT-5-REPORT.md` |

.mdc файлы (architecture, backend, frontend, business-communities, business-content, business-investing, business-merits) подгружаются автоматически Cursor'ом по glob patterns.

---

## Правила для Cursor (встроены в каждый промпт)

1. **Работай сериями**: backend → frontend, по задаче. Не всё сразу.
2. **После каждой серии**: запусти `pnpm build` (или аналог) и проверь ошибки.
3. **Не трогай** файлы, не упомянутые в PRD.
4. **Если непонятно** — прими решение, задокументируй в отчёте (⚠️).
5. **Отчёт** — в конце, в формате SPRINT-N-REPORT.md (см. шаблон ниже).

---

## Шаблон отчёта (единый для всех спринтов)

Файл: `docs/prd/projects/reports/SPRINT-N-REPORT.md`

```markdown
# Sprint N Report

## Статус
✅ Завершён / ⚠️ Частично / ❌ Заблокирован

## Что сделано
- [ ] (конкретные пункты из PRD/задач спринта)
- [ ] Билд проходит

## Решения, принятые по ходу
- ⚠️ (описание и причина)

## Не удалось / заблокировано
- ❌ (если есть)

## Файлы созданные/изменённые
- `path/to/file` — что сделано

## Чеклист для проверки (человеком)
- [ ] (критические сценарии для ручной проверки)
```

Для отдельных спринтов в промпте указаны **дополнительные блоки** (например, Sprint 3: «Где перехвачен withdraw», «Где merit resolver override»). Их нужно добавить в отчёт в дополнение к шаблону выше.
