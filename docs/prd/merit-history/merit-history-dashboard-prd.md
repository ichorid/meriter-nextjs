# PRD: Дэшборд «История заслуг» (Merit History Dashboard)

## Выбранный вариант (оптимальный, не MVP-урезанный)

**Единая панель над лентой** на страницах полной истории глобального кошелька:

1. **На каждой вкладке фильтра** (`Все`, `Передачи`, `Голоса и посты`, …) панель **пересчитывается** под активную вкладку: те же границы данных, что и у ленты (`meritHistoryCategory` / `MERIT_HISTORY_FILTER_KEYS`).
2. **Селектор периода** (общий для панели и спарклайна): **7 / 30 / 90 дней** от «сегодня» (календарные дни в TZ сервера или UTC — зафиксировать в TR). «Всё время» **не** входит в v1 (дорого и слабо сопоставимо с HR/инвест-нарративом без отдельной архивации).
3. **Первый ряд — KPI (4 показателя)** за выбранный период **в рамках текущей вкладки**:
   - **Приток** — сумма операций, которые в UI отображаются как зачисление на глобальный кошелёк (согласовать с `meritHistoryLedgerMultiplier` и знаком в ленте).
   - **Отток** — сумма списаний по модулю (положительное число для UX).
   - **Нетто** — приток − отток.
   - **Операций** — количество строк `transactions` в выборке.
4. **Второй ряд — спарклайн**: по **дням** внутри периода одна серия — **дневное нетто** (сумма знаковых изменений за день). Высота фиксированная (~48–64px), тёмная тема, без тяжёлых библиотек (**SVG или CSS-барчарт** в одном компоненте; новые зависимости типа Recharts **не** добавлять без отдельного обоснования).
5. **Только на вкладке «Все»** — под KPI+графиком **компактная таблица**: строки = **все категории кроме `all`** (те же бакеты, что у фильтров), столбцы = **нетто**, **оборот** (|приток| + |отток| по категории), **число операций** за тот же период. Это даёт HR и инвесторам **срез по каналам** без пролистывания ленты.

**Вне скоупа v1:** экспорт (CSV), цели/нормы, сравнение с прошлым периодом («vs предыдущие 30 дней»), drill-down в пост из таблицы категорий.

## Цель продукта

- **Инвесторы / аналитика:** быстро увидеть **масштаб и динамику** движений по интересующей категории (например «Инвестирование») за месяц.
- **HR:** на вкладке «Все» — **распределение вклада по типам активности** (таблица категорий) + динамика нетто по дням; на остальных вкладках — согласованный с фильтром срез.

## Контекст

- **Лента:** уже реализована (`MeritHistoryFeed`, `wallets.getTransactions`, infinite query, фильтры, i18n, enrichment).
- **Чужой пользователь:** полная лента при наличии права (`permissionCommunityId` / superadmin) — см. текущий код и `@.cursor/rules/business-merit-history.mdc`. Дэшборд обязан использовать **те же правила доступа** и те же `userId` / `permissionCommunityId`.
- **Данные:** агрегаты только из коллекции **`transactions`** для walletId глобального кошелька пользователя (как у ленты). Категория вкладки → тот же маппинг, что `meritHistoryReferenceTypeMatch` / `MERIT_HISTORY_FILTER_KEYS` (`wallet-transaction-history.ts`).

## Функциональные требования

- [ ] **FR-D1:** Панель дэшборда отображается **над** блоком вкладок+ленты на **`/meriter/profile/merit-transfers`** и **`/meriter/users/[id]/merit-history`** (и любых будущих экранах с тем же infinite-feed по `wallets.getTransactions`, если появятся).
- [ ] **FR-D2:** Селектор периода **7 | 30 | 90** дней; переключение периода **перезапрашивает** и панель, и (при необходимости согласования) ленту — **или** лента остаётся «вся история» без даты, а панель только по периоду — **выбрать один подход в TR-D3** и зафиксировать в AC (рекомендация PRD: **панель только по периоду**, лента **без** изменения диапазона дат в v1, чтобы не ломать UX бесконечной прокрутки).
- [ ] **FR-D3:** При смене **вкладки категории** обновляются KPI + спарклайн (+ таблица на «Все»).
- [ ] **FR-D4:** Пустой период: нули, плоский/пустой спарклайн, короткая подпись (i18n), без ошибок.
- [ ] **FR-D5:** Ошибка API / запрет: панель в error state, лента может оставаться доступной по отдельному запросу — не блокировать всю страницу одним упавшим запросом.
- [ ] **FR-D6:** Таблица категорий на «Все» — **скрывать строки с нулевыми операциями** во всех трёх столбцах (опционально строка «Прочее» если есть только uncategorized — уточнить по данным).

## Технические требования

- [ ] **TR-D1:** Новая процедура **`wallets.getMeritHistoryDashboard`** (или `meritHistory.*` в том же router-файле, но предпочтительно `wallets.*` для когерентности с `getTransactions`):
  - **Input:** `userId`, `category` (тот же enum, что фильтр ленты), `periodDays: 7 | 30 | 90`, `permissionCommunityId?: string` (обязателен по тем же правилам, что `getTransactions`, если `userId` ≠ caller).
  - **Output:** `{ kpis: { inflow, outflow, net, count }, series: Array<{ date: string, net: number }>, breakdown?: Array<{ category, net, grossVolume, count }> }` где `breakdown` заполняется **только** если `category === 'all'` (или отдельным флагом `includeBreakdown` — выбрать один вариант).
- [ ] **TR-D2:** Агрегация в MongoDB: `$match` по `walletId` + `createdAt` range + условие по `referenceType` через ту же логику, что фильтр ленты (вынести общий билдер `$match` или shared helper рядом с `meritHistoryReferenceTypeMatch`, **DRY**).
- [ ] **TR-D3:** Зафиксировать поведение ленты относительно периода (см. FR-D2 рекомендацию).
- [ ] **TR-D4:** Индексы: убедиться, что есть составной индекс под **`walletId` + `createdAt`** для `transactions`; при отсутствии — миграция/создание индекса в PR.
- [ ] **TR-D5:** Лимиты по нагрузке: один round-trip на смену вкладки/периода; кэш TanStack Query (`staleTime` разумный).
- [ ] **TR-D6:** i18n для подписей KPI, периода, пустого состояния, заголовка панели.

## Нефункциональные

- **Производительность:** p95 запроса дэшборда на типичном объёме — разумный бюджет (задать в PR, например < 300 ms на dev-сиде или логировать `explain`).
- **A11y:** период и вкладки — `aria-controls` / подписи; спарклайн — `aria-label` с кратким текстом тренда или табличное резюме для screen reader.

## Acceptance Criteria

- [ ] **AC-D1:** На каждой вкладке видны KPI и спарклайн, соответствующие выбранной категории и периоду.
- [ ] **AC-D2:** На «Все» дополнительно видна таблица категорий; суммы **согласованы** с тем, что пользователь увидит, просуммировав ленту за тот же период (в пределах округления `formatMerits`).
- [ ] **AC-D3:** Свой профиль и чужой (с правом) — одинаковый UI; без права — дэшборд **не** показывать или показывать только сообщение о доступе (как для ленты — **единая политика**).
- [ ] **AC-D4:** Нет регрессии `wallets.getTransactions` / ленты / прав.
- [ ] **AC-D5:** `pnpm lint && pnpm build` с корня репозитория.

## Этапы (для агента)

**Фаза 1 — Backend:** helper для `$match` по wallet + дате + категории; `getMeritHistoryDashboard`; unit/integration тест на известных `referenceType`.

**Фаза 2 — Web:** компонент панели (KPI + sparkline + условная таблица); wiring в `ProfileMeritTransfersClient` и `UserMeritHistoryClient`; состояние периода; запросы TanStack Query.

**Фаза 3 — Полировка:** пустые/ошибки, i18n, a11y, визуальное выравнивание с тёмной темой.
