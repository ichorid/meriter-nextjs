# PRD: История заслуг (Merit History)

## Цель

Заменить узкий раздел «Передачи заслуг» в профиле на **полноценный модуль «История заслуг»**: показывать все изменения заслуг на **глобальном счёте** пользователя (начисления и списания с понятным контекстом), дать **рубрикатор/фильтры** по типам операций и сохранить возможность работы со специализированными представлениями (община/проект, чужой профиль), не ломая существующую экономику.

## Контекст

- **Текущее состояние (web):** `/meriter/profile/merit-transfers` (`ProfileMeritTransfersClient`) показывает только **`meritTransfer.getByUser`** — peer-to-peer передачи из коллекции `merit_transfers`, вкладки «Входящие / Исходящие».
- **Текущее состояние (данные):** движения по кошелькам пишутся в **`transactions`** через `WalletService.addTransaction` с разными **`referenceType`** (голоса, выводы, комиссии, тапалка, инвестиции, `merit_transfer`, приветственные мериты и т.д.).
- **Критично для реализации:** процедура **`wallets.getTransactions`** вызывает `WalletService.getUserTransactions`, который в коде сейчас — **заглушка и возвращает пустой массив**. Полная история по строкам транзакций из API **недоступна** до доработки бэкенда.
- **Связанные модули:** `WalletService`, `meritTransfer` router, приоритетные хабы / `MeritResolverService`, инвестиции, тапалка, публикации/голоса, ивенты (`eventPostId` у передач).
- **Бизнес-правила (репозиторий):** `@.cursor/rules/business-merits.mdc`, `@.cursor/rules/business-merit-transfer.mdc`, `@.cursor/rules/business-investing.mdc`, `@.cursor/rules/business-tappalka.mdc`, `@.cursor/rules/business-content.mdc`.

## Требования

### Функциональные

- [ ] **FR-1:** В UI профиля заголовок и пункты навигации — **«История заслуг»** (вместо «Передачи заслуг» для глобальной ленты операций; см. FR-7 про отдельный поток P2P там, где уместно).
- [ ] **FR-2 (MVP объём данных):** В «Истории заслуг» отражаются **все движения по глобальному персональному кошельку** пользователя (`GLOBAL_COMMUNITY_ID`): зачисления и списания с суммой, знаком (приход/расход), временем и читаемым контекстом.
- [ ] **FR-3 (рубрикатор):** Фильтры по категориям операций, минимум:
  - прямые передачи между пользователями (`merit_transfer` / детали в `merit_transfers`);
  - голосование / комментарии с весом (по `referenceType` + обогащение);
  - инвестиции;
  - тапалка (`tappalka_show_cost`, `tappalka_reward`, …);
  - системные/прочие (приветственные мериты, комиссии постов/опросов/пересылок и т.д. — уточнить по полному списку `referenceType` в коде);
  - опционально **«Все»** без фильтра.
- [ ] **FR-4:** Единая хронологическая лента (новые сверху), бесконечная прокрутка или пагинация, состояния loading/empty/error.
- [ ] **FR-5:** Для каждой записи: сумма, направление, время; вторичная строка — контекст (сообщество/проект, пост по `referenceId`, контрагент для P2P). Где нельзя резолвить — безопасный fallback без утечки приватных данных.
- [ ] **FR-6:** Где есть пост/пользователь/сообщество — ссылки на существующие маршруты (`routes.*`).
- [ ] **FR-7 (сценарии):**
  - **Профиль → История заслуг:** полная история глобального счёта (FR-2).
  - **Контекст общины/проекта** (`…/merit-transfers`): либо узкий список передач в контексте, либо фильтр «только peer transfers здесь» — зафиксировать один подход (без двух конкурирующих источников истины).
  - **Чужой профиль** (`/meriter/users/[id]/merit-transfers`): явно задать политику приватности в AC.

### Технические

- [ ] **TR-1:** Реализовать **`WalletService.getUserTransactions`** и/или новую процедуру (`wallets.*` или `meritHistory.*`): чтение `transactions` для wallet пользователя с `GLOBAL_COMMUNITY_ID`, пагинация, сортировка по дате, фильтр по категориям (маппинг `referenceType` → категория UI).
- [ ] **TR-2:** Зафиксировать каталог `referenceType` (константа/enum + таблица для UI), пройти все `addTransaction` и покрыть маппинг.
- [ ] **TR-3:** Обогащение строк (заголовок поста, имя сообщества) — батч-резолверы; без N+1 без лимитов.
- [ ] **TR-4:** Не оставлять «тихий» пустой ответ у `wallets.getTransactions` при наличии данных в БД.
- [ ] **TR-5:** Источник для P2P в общей ленте: либо только wallet lines, либо слияние с `merit_transfers` — **без дублирования строк**.
- [ ] **TR-6:** i18n для заголовков, фильтров, категорий.

### Cursor должен проверить в коде

- [ ] **CHECK-1:** `getUserTransactions` и вызов из `wallets.router.ts`.
- [ ] **CHECK-2:** Полный список `referenceType` из всех `addTransaction`.
- [ ] **CHECK-3:** `MeritResolverService` vs отображение «глобально» в UI.
- [ ] **CHECK-4:** Права доступа к истории (self vs other users).
- [ ] **CHECK-5:** Маршруты `profileMeritTransfers`, `communityMeritTransfers`, `userProfileMeritTransfers`.

## Этапы реализации

**Фаза A — Backend:** реальная выборка транзакций глобального кошелька + total/count; маппинг категорий; тесты маппинга и пагинации.

**Фаза B — API контракт:** стабильный input (page/cursor, limit, categories), обогащённые DTO для списка.

**Фаза C — Web:** переименование, рубрикатор, лента; обновление `ProfileMeritTransfersClient`.

**Фаза D:** Согласование страниц community/project/user merit-transfers и терминологии.

**Фаза E:** A11y, перформанс списка, скелетоны.
